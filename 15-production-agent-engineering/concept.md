# Agent 评估与可观测性：通往生产的最后一公里

> 制造一个 Agent 很容易，但维护一个可靠的 Agent 很难。本阶段探讨如何量化 AI 的表现，并在它“抽风”时第一时间抓到它。

---

## 一、 核心痛点：不可测则不可管

LLM 的随机性使得 Agent 的迭代非常危险。如果你没有一套自动化评估系统，你每改一行代码，都在进行一场“豪赌”。

### 1. 幻觉 (Hallucination)
Agent 引用了知识库中根本不存在的事实。

### 2. 回复漂移 (Drift)
升级了模型版本或修改了 Prompt，导致原本能回答的问题现在答不出来了。

---

## 二、 评估方法论：让 AI 审判 AI

### 1. RAGAS 评估框架 (针对 RAG)
它提出了四个核心维度，每个维度都是一个 0-1 的得分：
*   **Faithfulness (忠实度)**：答案是否来源于检索到的文档？（防幻觉）
*   **Answer Relevance (答案相关性)**：答案是否真的解决了用户的问题？（防跑题）
*   **Context Precision (检索精度)**：检索到的文档是不是真的有用？（评测 LlamaIndex 效果）
*   **Context Recall (检索召回率)**：是否找全了所有必要的资料？

### 2. LLM-as-a-Judge
用 GPT-4o 这种顶级模型，根据你预设的评分准则（Rubric），给其他 Agent 的输出打分。这虽然贵，但非常接近人类专家的判断。

---

## 三、 可观测性 (Observability)

你不能只看最终结果，你必须看清 **Trace（追踪）**。

### 1. 什么是 Trace？
一个复杂的 Agent 请求可能包含 10 次 LLM 调用、3 次数据库检索。Trace 能记录下：
*   每次调用的 **Prompt**。
*   每次调用的 **Latency (耗时)**。
*   每次调用的 **Token 消耗**。

### 2. 核心工具
*   **Langfuse / LangSmith**：Agent 调试和监控的神器。
*   **Arize Phoenix**：开源、本地化，支持 RAG 深度分析。

---

## 四、 护栏 (Guardrails)

在数据输出给用户前，增加一层“过滤器”：
*   **内容审计**：检查是否有暴力、歧视内容。
*   **PII 脱敏**：防止 Agent 泄露用户的电话、地址。
*   **结构校验**：确保 Agent 输出的是合法的 JSON。

---

## 五、 本阶段学习目标

1.  理解 RAGAS 的四个核心评价指标。
2.  分析如何使用 LLM 进行自动化评审。
3.  了解 LangSmith 等工具在生产环境中的位置。
