## 当前基线

- 稳定提交：`ccb1f24 feat(content-team): improve report quality workflow`
- 基线说明：已包含质量门禁、Writer 本地自修、Researcher 来源重试、引用闭环、结构检查和单元测试。

## 目标样本质量特征

参考样本：`reports/final_report_20260512_144407.md`

该报告虽然也存在“综合案例较多、公开可核验案例不足”的证据弱点，但在阅读体验、结构节奏和信息密度上明显优于后续多次生成结果，应作为阶段 1 评估工具和阶段 2/3 prompt 优化的重要目标样本。

可量化特征：

| 指标 | `144407` 表现 | 后续评估目标 |
| :--- | :--- | :--- |
| 主章节数量 | 9 个 | 稳定在 8-10 个，优先 9-10 个 |
| 三级小节数量 | 19 个 | 15-22 个，避免过少或标题堆叠 |
| 列表项数量 | 37 个 | 至少 25 个，关键章节应有清单增强信息密度 |
| 引用来源数量 | 14 条 | 至少 10 条，且区分强弱来源 |
| 小节平均厚度 | 约 115 字符单位 | 平均不低于 100，薄小节占比低 |
| 过薄小节 | 仅 1 个低于 80 字符单位 | 低于 80 的小节不超过 2 个 |
| 案例覆盖 | 5 个应用场景案例章 | 案例章应覆盖客户、运营、决策、组织、风控等不同场景 |
| 失败/挑战内容 | 多处穿插失败教训和挑战 | 不能只写成功案例，必须形成正反对照 |

结构特征：
- 前两章先建立技术背景、价值和挑战，后续案例不会直接空降。
- 第三至第七章形成稳定但不机械的案例节奏：成功案例、失败或挑战、横向或纵向分析。
- 第八至第九章承接案例，转向实施路径、最佳实践、失败教训总结和高管建议。
- 小节标题有编辑判断，例如“挑战与教训”“横向对比”“纵向分析”“给企业高管的建议”，不是机械的 `1.1/1.2/1.3`。

阅读体验特征：
- 格式有变化：段落、项目符号、案例分析、对比判断和建议清单交替出现。
- 每个小节都有自己的内部结构，不是“标题 + 单段短文”。
- 成功案例和失败教训形成节奏，读者能看到价值、边界和落地条件。
- 后半部分不是简单总结，而是给出路径、风险边界和面向决策者的建议。

需要吸收但不能照搬的问题：
- `144407` 大量使用综合案例，后续优化必须补上来源强弱标注和证据边界。
- 不能为了证据谨慎把文章写干，应保留其编辑节奏和格式丰富度。
- 评估工具应同时奖励“可读性与结构厚度”，也惩罚“弱来源支撑核心结论”。

### [ ] 阶段 2：降低 Writer 输出随机波动

目标：稳定报告的编辑节奏和格式丰富度，不改变整体流程。

任务：
- 调整 Writer prompt，强调“少拆小节、写厚小节”。
- 要求案例章节有充分展开，而不是短标题堆叠。
- 要求至少包含比较表、实施路径清单、风险/证据边界段落中的若干格式变化。
- 避免机械生成 `1.1/1.2/1.3` 式结构。

完成标准：
- 新生成报告的小节平均长度、列表数量、格式变化不低于 `094340` 的合理区间。
- 新生成报告的案例章节节奏接近 `144407`，即成功案例、失败/挑战、横向/纵向分析能自然交替。
- 结构门禁不频繁触发。
- 人工阅读感受不低于 `094340`。

### [ ] 阶段 3：提高 PM 大纲质量

目标：让大纲更像主编规划，而不是随机章节列表。

任务：
- 精简 PM prompt，避免堆过多门禁式要求。
- 强调读者决策、核心问题、章节角色和证据策略。
- 稳定主章节数量，优先 9-10 个主章节。
- 避免案例章节过泛、过碎或证据不可核验。

完成标准：
- 多次生成大纲的章节数量和结构角色更稳定。
- 大纲能自然支撑比较、风险、实施路径等后半段内容。
- 大纲能复现 `144407` 的“背景-价值挑战-多场景案例-实施路径-高管建议”主线，但不复用其具体标题。
- 不明显降低 LLM 的主编式判断质量。

### [ ] 阶段 4：将门禁原则前移到角色提示词

目标：减少门禁触发，把质量要求在源头传递给 PM、Researcher、Writer、Reviewer。

任务：
- 将确定性门禁提炼成短原则，而不是长规则。
- PM 负责避免弱证据方向成为核心章节。
- Researcher 负责标注来源强弱和案例可核验性。
- Writer 负责弱来源降级表达，避免用弱来源支撑核心 ROI。
- Reviewer 只对影响可信度、结构或事实边界的问题拒绝。

完成标准：
- 门禁触发次数下降。
- `review_count` 不再主要消耗在结构问题上。
- 报告质量提升来自首轮生成，而不是反复返工。

### [ ] 阶段 5：来源质量专项优化

目标：减少因厂商博客、供应商宣传、综合案例支撑核心结论而被拒绝。

任务：

- 优先召回官方报告、公司案例页、咨询机构 PDF、权威媒体转述。
- 对案例章节要求至少一个较高质量来源；否则标记为趋势观察或证据边界。
- 避免厂商博客 ROI 成为核心结论。

完成标准：
- 高质量来源占比提升。
- Reviewer 因“核心案例弱来源”拒绝的频率下降。
- 报告仍保持可读性，不因谨慎而变干。

### [ ] 阶段 7：Token 成本与缓存机制优化

目标：降低搜索、写作、评审和返工带来的重复 token 消耗，并提高可复用中间产物的命中率。

当前问题：
- Researcher 每次按章节重新搜索和提炼，Tavily 结果与 LLM 提炼结果没有持久缓存。
- Writer 每轮都会接收完整研究素材、来源索引、历史摘要和质量反馈，prompt 较长且变化大。
- Reviewer 每轮接收完整正文，语义评审成本随报告长度线性增长。
- 门禁失败会触发重写，虽然 Writer 本地自修减少了 `review_count` 消耗，但仍增加 LLM 调用。
- DeepSeek 的 prompt cache 命中率低，原因可能包括动态时间戳、长上下文顺序变化、反馈文本变化、来源排序变化和完整正文每轮变化。

分析任务：
- 统计每个节点的调用次数、输入字符数、输出字符数、耗时和重试次数。
- 区分不可避免的新生成成本与可缓存的重复成本。
- 找出 prompt 中频繁变化但不影响语义的字段，例如日志摘要、来源顺序、动态反馈格式。
- 对比一次完整运行中 PM、Researcher、Writer、Reviewer 的成本占比。

缓存设计候选：
- **搜索结果缓存**：以搜索 query 的规范化文本为 key，缓存 Tavily 返回结果，设置 TTL 或手动失效。
- **研究素材缓存**：以 `topic + section + search_result_hash + feedback_hash` 为 key，缓存 `ResearchMaterial`。
- **来源排序缓存**：以 `ResearchReport` 的 sources hash 为 key，缓存 ranked source records。
- **报告评估缓存**：确定性评估不需要 LLM，可直接复用本地计算，不进入模型调用。
- **Prompt 稳定化**：将长而稳定的 rubric、格式规范、角色说明固定在 system prompt 或常量中，减少动态拼接；动态内容集中放在较短的 user prompt。
- **差量评审**：Reviewer 在返工轮优先评审变化摘要和上轮问题，而不是每次完整重读全部正文；仅最终轮或风险高时读全文。
- **资料摘要层**：Writer 不直接消费所有 raw_data，而是消费经过压缩的“章节证据卡片”，保留来源编号、强弱、可用结论和禁用结论。

完成标准：
- 增加成本观测日志，能输出每个节点的调用次数和字符量。
- 至少实现 Tavily 搜索结果缓存或 ResearchMaterial 缓存中的一种。
- 不降低报告质量，不引入过期资料误用。
- 同一主题重复运行时，搜索或研究提炼成本有可观下降。

风险与约束：
- 缓存不能掩盖资料过期问题；涉及“2026 最新”等主题时需要可配置失效策略。
- 不能为了命中缓存牺牲提示词的清晰度和角色判断质量。
- Reviewer 差量评审可能漏掉全文一致性问题，因此仍需保留最终全文评审。

建议提交粒度：
- `chore(content-team): add agent cost instrumentation`
- `perf(content-team): cache research search results`
- `perf(content-team): cache extracted research materials`
- `perf(content-team): compact writer evidence context`
