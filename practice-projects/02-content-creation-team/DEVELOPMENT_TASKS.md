## 阶段性暂停结论

2026-05-14 暂停继续优化内容生成主流程。

本项目已经验证了多 Agent 报告生成的主要难点：高质量报告不是简单通过“PM 大纲 + Researcher 搜索 + Writer 写作 + Reviewer 评审”就能稳定产出。每次修复都可能引入新的质量波动；有些改动能解决一个样本的问题，却会让另一个样本变差。持续追加补丁会让角色代码、状态流转和返工逻辑越来越复杂，最终难以判断系统为什么成功或失败。

最近一次真实运行生成 `reports/final_report_20260514_112128.md`，流程最终通过 Reviewer：
- 字数：4723。
- 引用来源：15 条。
- 评审次数：3。
- 确定性评估：`100` 分，结构指标优于多个历史样本。

但人工复核后仍不应把它视为完全可外发报告：
- 弱来源仍支撑了部分核心数据，例如行业访谈、博客或转载数据。
- 存在引用与具体句子支撑关系可能错配的问题。
- 综合案例和厂商自述虽已被标注降级，但仍不能替代公开可核验案例。
- 现有确定性评估能识别结构和显性证据边界，却不能判断引用是否真正支撑对应论断。

因此当前策略是：停止继续增加在线 Agent 分支和 prompt 补丁，保留已经完成的简化与模块拆分。后续若重启，应优先做离线事实核查、引用支撑关系检查和人工评审辅助，而不是继续扩大主流程复杂度。

## 关于 `144407` 样本的复盘

`reports/final_report_20260512_144407.md` 是一次测试验证中生成的高质量样本。它在阅读体验、结构节奏和案例推进上明显优于后续大量版本，但当时的大纲、提示词、搜索结果、上下文状态和生成条件没有完整保存，因此它不能被当作稳定能力，只能作为一个不可完全复现的优质样本。

后续优化长期围绕 `144407` 倒推目标，出现了明显的过拟合风险：
- 每发现一个失败样本，就增加一条 prompt 约束、门禁规则或返工分支。
- 某些规则能修复当前样本，却会破坏另一个维度，例如文章节奏、案例承载力或证据表达自然度。
- 动态大纲每次都会改变 Researcher 和 Writer 的任务边界，导致同一套 prompt 难以稳定复现相同质量。
- 代码逐渐承载历史补丁记忆，而不是清晰、可解释的设计。

因此，后续如果重新尝试，不应直接继续优化完整多 Agent 主流程。更可控的路径是：
1. 固定大纲、固定搜索材料、固定引用清单和目标样本，先让 Writer 在固定输入下复现 `144407` 级别的阅读质量。
2. 保存每次高质量样本对应的 prompt、输入材料、模型参数和评估结果，形成可复现实验资产。
3. 先离线验证单个角色能力，再逐步放开 PM 动态大纲和 Researcher 动态搜索。
4. 把“引用是否真正支撑对应句子”作为独立事实核查问题处理，不再依赖 Reviewer 的泛化判断。

核心经验：不要把一次好结果误判为系统能力。只有能在固定输入下稳定复现，才有资格继续扩大到动态多 Agent 流程。

- **不要把一次好结果当作系统能力。** 如果不能复现，它只是样本，不是能力。
- **动态大纲是质量波动的最大来源之一。** 每次换大纲，Researcher 和 Writer 实际上都在做新任务。
- **Prompt 优化很容易过拟合历史失败样本。** 尤其是写作风格、证据边界、案例结构这些目标本身存在张力。
- **Reviewer 通过不等于报告质量稳定。** 它只能降低明显错误，不能保证引用支撑和文章质感。
- **复杂流程会吞掉可解释性。** 当节点、返工、门禁、降级、增强轮越来越多，就很难判断是哪一步让质量变好或变差。

## 目标样本质量特征

参考样本：`reports/final_report_20260512_144407.md`

该报告虽然也存在“综合案例较多、公开可核验案例不足”的证据弱点，但在阅读体验、结构节奏和信息密度上明显优于后续多次生成结果，应作为阶段 1 评估工具和阶段 2/3 prompt 优化的重要目标样本。

可量化特征：

| 指标 | `144407` 表现 | 后续评估目标 |
| :--- | :--- | :--- |
| 主章节数量 | 9 个 | 稳定在 8-10 个，优先 9-10 个 |
| 三级小节数量 | 19 个 | 15-22 个，避免过少或标题堆叠 |
| 列表项数量 | 37 个 | 至少 25 个，关键章节应有清单增强信息密度 |
| 引用来源数量 | 14 条 | 至少 10 条，且区分强弱来源 |
| 小节平均厚度 | 约 115 字符单位 | 平均不低于 100，薄小节占比低 |
| 过薄小节 | 仅 1 个低于 80 字符单位 | 低于 80 的小节不超过 2 个 |
| 案例覆盖 | 5 个应用场景案例章 | 案例章应覆盖客户、运营、决策、组织、风控等不同场景 |
| 失败/挑战内容 | 多处穿插失败教训和挑战 | 不能只写成功案例，必须形成正反对照 |

结构特征：
- 前两章先建立技术背景、价值和挑战，后续案例不会直接空降。
- 第三至第七章形成稳定但不机械的案例节奏：成功案例、失败或挑战、横向或纵向分析。
- 第八至第九章承接案例，转向实施路径、最佳实践、失败教训总结和高管建议。
- 小节标题有编辑判断，例如“挑战与教训”“横向对比”“纵向分析”“给企业高管的建议”，不是机械的 `1.1/1.2/1.3`。

阅读体验特征：
- 格式有变化：段落、项目符号、案例分析、对比判断和建议清单交替出现。
- 每个小节都有自己的内部结构，不是“标题 + 单段短文”。
- 成功案例和失败教训形成节奏，读者能看到价值、边界和落地条件。
- 后半部分不是简单总结，而是给出路径、风险边界和面向决策者的建议。

需要吸收但不能照搬的问题：
- `144407` 大量使用综合案例，后续优化必须补上来源强弱标注和证据边界。
- 不能为了证据谨慎把文章写干，应保留其编辑节奏和格式丰富度。
- 评估工具应同时奖励“可读性与结构厚度”，也惩罚“弱来源支撑核心结论”。

### [!] 阶段 7：Token 成本与缓存机制优化

目标：降低搜索、写作、评审和返工带来的重复 token 消耗；本阶段以成本观测为主，不再继续投入 Tavily 搜索结果缓存。

当前问题：
- Researcher 每次按章节重新搜索和提炼，Tavily 结果与 LLM 提炼结果没有持久缓存。
- Writer 每轮都会接收完整研究素材、来源索引、历史摘要和质量反馈，prompt 较长且变化大。
- Reviewer 每轮接收完整正文，语义评审成本随报告长度线性增长。
- 门禁失败会触发重写，虽然 Writer 本地自修减少了 `review_count` 消耗，但仍增加 LLM 调用。
- DeepSeek 的 prompt cache 命中率低，原因可能包括动态时间戳、长上下文顺序变化、反馈文本变化、来源排序变化和完整正文每轮变化。

分析任务：
- 统计每个节点的调用次数、输入字符数、输出字符数、耗时和返工次数。
- 区分不可避免的新生成成本与可缓存的重复成本。
- 找出 prompt 中频繁变化但不影响语义的字段，例如日志摘要、来源顺序、动态反馈格式。
- 对比一次完整运行中 PM、Researcher、Writer、Reviewer 的成本占比。

缓存设计候选：
- **搜索结果缓存**：已评估并放弃，后续不再关注。完整 query 缓存命中率过低；语义桶缓存容易污染资料召回，会反向伤害来源质量专项优化，不适合当前学习项目。
- **研究素材缓存**：以 `topic + section + search_result_hash + feedback_hash` 为 key，缓存 `ResearchMaterial`。
- **来源排序缓存**：以 `ResearchReport` 的 sources hash 为 key，缓存 ranked source records。
- **报告评估缓存**：确定性评估不需要 LLM，可直接复用本地计算，不进入模型调用。
- **Prompt 稳定化**：将长而稳定的 rubric、格式规范、角色说明固定在 system prompt 或常量中，减少动态拼接；动态内容集中放在较短的 user prompt。
- **差量评审**：Reviewer 在返工轮优先评审变化摘要和上轮问题，而不是每次完整重读全部正文；仅最终轮或风险高时读全文。
- **资料摘要层**：Writer 不直接消费所有 raw_data，而是消费经过压缩的“章节证据卡片”，保留来源编号、强弱、可用结论和禁用结论。

完成标准：
- 增加成本观测日志，能输出每个节点的调用次数和字符量。
- 当前阶段先不实现缓存，保留成本观测和独立日志；后续只考虑 URL 证据缓存、来源质量分类缓存或案例候选缓存这类低污染风险方案。
- 不降低报告质量，不引入过期资料误用。
- 同一主题重复运行时，可以通过日志分析主要成本来源，为后续缓存策略提供依据。

当前进展：
- 已新增 `utils/cost_utils.py`，在 PM、Researcher、Writer、Reviewer 的 LLM 调用处记录输入字符、输出字符、耗时和缓存命中状态；Summarizer 已从主流程移除。
- 已尝试 Tavily 搜索缓存，但因 PM 大纲动态生成导致精确 query 命中率低；语义稳定 key 又会带来搜索结果污染风险，已按学习项目“保持简单”的原则撤回。该方向收益不足，后续不再作为优化任务。
- 已新增 `.gitignore` 忽略 `logs/`，避免把运行日志提交到仓库。
- 已支持独立文件日志：默认输出到 `logs/run_YYYYMMDD_HHMMSS.log`，可通过 `CONTENT_TEAM_LOG_DIR` 和 `CONTENT_TEAM_LOG_TO_FILE` 配置。
- 2026-05-14 暂停结论：当前主要瓶颈不是 token 成本，而是事实可信度和引用支撑关系。继续做缓存可能降低成本，但不会解决报告质量的核心问题，因此暂停。
