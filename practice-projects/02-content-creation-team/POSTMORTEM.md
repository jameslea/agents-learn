# 项目 B 复盘：内容创作团队为什么不稳定

> 复盘日期：2026-05-15
> 状态：暂停继续优化在线主流程，保留为多 Agent 失败模式样本库。

## 背景

项目 B 的目标是用多角色协作生成结构化深度报告：

```text
PM -> Researcher -> Writer -> Reviewer
```

项目最终实现了结构化 artifact、质量门禁、引用校验、人类断点、报告评估工具和多轮真实运行。但阶段性结论是：当前系统适合作为学习项目和流程实验样本，不应视为稳定的报告生产系统。

最近一次真实运行 `reports/final_report_20260514_112128.md` 通过了 Reviewer，结构和可读性较好，但人工复核仍发现弱来源支撑核心数据、引用与句子支撑关系可能错配、综合案例缺少可追溯证据等问题。

## 最重要的结论

多 Agent 报告生成的难点不在“让多个角色说话”，而在：

- 证据是否可靠。
- 引用是否支撑对应论断。
- 大纲、材料、写作风格是否能稳定复现。
- Reviewer 是否真的能发现关键问题。
- 每一次好结果是否能被保存为可复现实验资产。

项目 B 暴露的问题说明：如果没有离线评估和事实核查，继续增加在线 Agent 分支只会让系统更复杂、更难解释。

## 失败模式

### 1. 一次好结果被误判为系统能力

`reports/final_report_20260512_144407.md` 是目前阅读体验最好的样本之一，但当时的大纲、搜索结果、prompt、模型参数和完整中间状态没有保存。

因此它只能说明“某次采样很好”，不能证明系统稳定具备该能力。

后续围绕该样本倒推优化，容易产生过拟合：

- 为一个失败样本增加一条 prompt 规则。
- 修复结构问题后牺牲文章节奏。
- 强化证据边界后报告变干。
- 门禁越来越多，但质量波动仍存在。

### 2. 动态大纲放大了下游不稳定

PM 每次生成的大纲不同，Researcher 和 Writer 实际面对的是新任务。

这会导致：

- 搜索词变化。
- 来源集合变化。
- 写作节奏变化。
- Reviewer 拒绝原因变化。

在没有固定输入实验前，直接优化完整链路，很难判断变化来自 PM、Researcher、Writer 还是 Reviewer。

### 3. Researcher 是质量瓶颈

报告质量取决于材料质量。Researcher 如果只找到弱来源、匿名案例、厂商白皮书或二手转载，Writer 无法凭 prompt 写出真正可外发的报告。

典型问题：

- 可核验案例不足。
- 趋势材料被写成案例证据。
- 弱来源支撑核心结论。
- 综合案例和真实案例边界不清。
- 横向分析章节没有复用前文证据地图。

项目中曾尝试跨章节 ResearchPlan 和 EvidenceMap，但在线接入后复杂度明显上升，且没有稳定解决核心问题，因此已回退。

### 4. Reviewer 通过不等于事实可靠

Reviewer 可以识别结构问题、缺引用、格式错误和部分证据边界问题，但很难稳定判断：

- 引用是否真的支撑当前句子。
- 来源中的数字是否被正确转述。
- 厂商自述是否被过度外推。
- 综合案例是否被包装成真实案例。

这类问题需要单独的引用支撑关系检查，而不是继续依赖泛化 Reviewer。

### 5. 流程补丁吞掉可解释性

多轮优化后，系统出现了典型复杂化风险：

- 节点越来越多。
- prompt 约束越来越长。
- 门禁和返工分支越来越多。
- 历史失败样本以隐式规则留在代码里。
- 真实运行失败后，很难判断是哪一处导致退化。

这说明下一步不应继续堆在线流程，而应先把评估、复现和事实核查做成独立能力。

## 已经有价值的沉淀

虽然报告生产能力不稳定，但项目 B 产出了很有价值的工程资产：

- `sop_artifacts.py`：结构化中间产物。
- `utils/report_evaluation.py`：确定性报告评估。
- `utils/outline_evaluation.py`：大纲质量评估。
- `utils/quality.py`：引用、来源、章节和字数门禁。
- `utils/source_quality.py`：来源质量判断。
- `reports/`：大量 rejected/final 样本，可作为失败模式数据集。
- `DEVELOPMENT_TASKS.md`：完整优化轨迹和阶段性判断。
- `RESEARCH_PLAN_TASKS.md`：跨章节研究计划的设计和回退记录。

这些资产比当前主流程本身更重要。后续应优先把它们转成评估集、事实核查工具和方法论样本。

## 后续重启条件

只有满足以下条件，才建议继续优化项目 B：

1. 固定一个目标主题、目标大纲、搜索材料和引用清单。
2. 保存每次运行的 prompt、模型参数、输入 artifact 和输出 artifact。
3. 先证明 Writer 能在固定输入下稳定复现高质量报告。
4. 建立引用支撑关系检查，至少能抽样判断“句子 -> 引用”是否成立。
5. 将人工事实核查结果写回评估集。
6. 再逐步放开 PM 动态大纲和 Researcher 动态搜索。

不满足这些条件时，不应继续给主流程加 prompt 补丁。

## 可转化为项目 D 的经验

项目 D 的自愈运维 Agent 应吸收这些经验：

| 项目 B 经验 | 项目 D 对应约束 |
|-------------|----------------|
| Reviewer 通过不等于质量可靠 | 修复成功必须由测试或命令结果验证 |
| 弱来源会污染后续写作 | 错误分类必须压缩并保留关键 traceback |
| 动态大纲导致输入漂移 | challenge task 和验收命令必须固定 |
| 多轮补丁降低可解释性 | 每轮 patch、错误、判断和结果必须 trace |
| 复杂返工容易失控 | 最大重试、超时和失败终止必须硬编码 |
| 角色边界不能只靠 prompt | Coder、Executor、Reviewer 必须有不同权限 |

项目 D 的重点应是“受控自愈循环”，不是“多个 Agent 聊天修代码”。

## 下一步建议

1. 保留当前项目 B 主流程，不继续扩展。
2. 把 `reports/` 中的 rejected 样本整理为失败模式评估集。
3. 单独实现引用支撑关系检查工具，再决定是否回到 B。
4. 进入项目 D 前，先用 `docs/concepts/agent-design-methodology.md` 作为设计检查表。
