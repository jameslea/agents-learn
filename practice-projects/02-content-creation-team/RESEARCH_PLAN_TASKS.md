# Cross-Section Research Plan Tasks

本文档记录 Researcher 从“逐章独立搜索”升级为“跨章节研究计划”的独立设计任务。该方向实现复杂度高，先作为阶段 8 单独跟踪，不直接混入阶段 5 的搜索词和来源质量优化。

## 背景问题

当前 Researcher 的实际流程是：

1. 按大纲章节顺序遍历。
2. 每个章节单独构造搜索词。
3. 每个章节单独调用 Tavily。
4. 每个章节只基于本章搜索结果调用 LLM 提炼 `ResearchMaterial`。
5. 最后把所有章节材料合并为 `ResearchReport`。

这个流程虽然简单，但隐含假设是“章节之间可以独立调研”。真实报告并不满足这个假设：

- 横向比较章依赖前面案例章、技术章和来源强弱，而不是从零搜索。
- 风险与证据边界章应综合前文材料中的弱来源、厂商自述、匿名案例和归因限制。
- 实施路径章需要从案例、风险、ROI、部署模式中归纳行动建议，不应只搜索“实施路径”。
- 结论和展望章通常是全篇综合，不应引入大量新的无关搜索结果。
- 案例章之间也有关联：如果某类案例公开证据不足，后续案例搜索应更强地转向官方案例、年报、新闻稿或可核验企业名称。

这也是 `reports/rejected_report_20260513_184444.md` 暴露出的根因之一：Researcher 已经在多个章节发现可核验证据不足，但这些信息没有在调研阶段形成跨章节证据地图，Writer 只能在最后被动处理。

## 目标

把 Researcher 从“章节级独立搜索器”改造成“按研究计划采集证据，再按章节组织材料”的流程。

目标不是增加更多搜索，而是提高搜索和提炼的依赖意识：

- 先采集一手证据和案例候选。
- 再构建可复用的证据地图。
- 后续综合章节优先使用前文证据，而不是重复搜索。
- 对需要补搜的章节，使用前文证据缺口生成更准确的搜索词。
- 让 Writer 收到的 `ResearchReport` 更像一套互相支撑的材料，而不是若干孤立摘要。

## 拟议设计

### 1. ResearchPlan

新增确定性的研究计划结构，先不引入额外 LLM 调用。

每个章节根据标题和位置分类：

- `search_primary`：必须独立搜索，主要产出原始事实、案例和强来源。
- `hybrid`：需要搜索，但搜索词和提炼需依赖前文材料。
- `synthesis`：主要基于前文材料归纳，默认不强制 Tavily 搜索，必要时少量补搜。

建议初始映射：

| 章节类型 | 搜索模式 | 依赖 |
| :--- | :--- | :--- |
| 背景/技术基础/市场数据 | `search_primary` | 无或仅依赖主题 |
| 具体案例/应用实践 | `search_primary` | 可参考前面证据缺口 |
| 横向比较/模式归纳 | `synthesis` | 案例章、技术章、来源质量 |
| 风险/挑战/治理 | `hybrid` | 案例章、技术章、证据缺口 |
| 证据边界/可验证性 | `synthesis` | 全部 `case_candidates`、`source_quality` |
| ROI/价值量化 | `hybrid` | 案例章、市场数据、来源质量 |
| 实施路径/决策框架 | `synthesis` | 案例章、风险章、价值评估 |
| 结论/展望 | `synthesis` | 全文材料 |

### 2. EvidenceMap

在完成 `search_primary` 章节后，构建一个轻量证据地图：

- 可写案例候选：`is_writable_case=true`
- 不可写案例候选：匿名、综合、厂商自述、趋势观察
- 高质量来源：tier_1/tier_2
- 弱来源和降级原因：tier_3、博客、聚合资讯、厂商软文
- 可复用事实：技术能力、市场数据、风险限制、ROI 边界
- 证据缺口：缺企业名称、缺独立验证、缺量化数据、缺失败教训

EvidenceMap 先作为内部 prompt 上下文，不一定马上扩展 `sop_artifacts.py` 的公开 schema。

### 3. 分阶段调研流程

建议将 `Researcher.conduct_research()` 从 list comprehension 改为显式阶段：

1. 生成 `ResearchPlan`。
2. 执行 `search_primary` 章节。
3. 从已完成材料构建 EvidenceMap。
4. 执行 `hybrid` 章节：搜索词带上 EvidenceMap 的关键词和缺口。
5. 执行 `synthesis` 章节：默认使用 EvidenceMap 和相关章节摘要，必要时补搜。
6. 合并材料并保持原始大纲顺序。

### 4. Prompt 调整

对依赖章节的素材提炼 prompt 增加：

- “可复用前文证据摘要”
- “禁止重复引入弱来源作为核心证据”
- “如果前文证据不足，应输出证据边界，不要构造新案例”
- “横向比较和实施路径必须引用前文案例/来源，而不是孤立生成”

## 非目标

- 暂不增加 PM 大纲字段。
- 暂不增加新的 LLM 规划节点。
- 暂不把所有章节改成复杂 DAG。
- 暂不实现 Tavily 缓存。
- 暂不要求 synthesis 章节完全不搜索；先允许配置化补搜。

## 实施拆分

### [ ] 8.1 研究计划分类器

任务：
- 新增 `ResearchSectionPlan` 或等价内部结构。
- 基于现有 `classify_research_section()` 扩展出 `search_mode` 和 `depends_on_roles`。
- 输出调研计划日志，便于观察每章是 `search_primary`、`hybrid` 还是 `synthesis`。

验证：
- 单测覆盖横向比较、风险、实施路径、证据边界和案例章的分类。
- 不改 Tavily 调用行为。

### [ ] 8.2 EvidenceMap 构建

任务：
- 从已完成 `ResearchMaterial` 中提取案例候选、来源强弱和证据缺口。
- 生成可放入 prompt 的短摘要，控制长度，避免 Writer/Researcher prompt 膨胀。

验证：
- 单测覆盖可写案例、不可写案例、强来源、弱来源、证据缺口摘要。
- EvidenceMap 不改变现有 `ResearchReport` schema。

### [ ] 8.3 Hybrid/Synthesis 章节上下文注入

任务：
- `hybrid` 章节搜索时加入 EvidenceMap 中的关键实体、缺口和来源类型。
- `synthesis` 章节提炼时注入相关前文材料摘要。
- 横向比较、证据边界、实施路径优先使用前文材料。

验证：
- 单测确认横向比较 prompt 包含案例章摘要。
- 单测确认实施路径 prompt 包含风险和案例摘要。
- 单测确认证据边界 prompt 包含 `case_candidates` 和来源质量摘要。

### [ ] 8.4 搜索调用控制

任务：
- 允许 `synthesis` 章节默认不调用 Tavily，只基于 EvidenceMap 生成材料。
- 对需要外部 benchmark 的 synthesis 章节保留补搜选项。
- 增加日志统计每类章节的 Tavily 调用次数。

验证：
- 单测确认 synthesis 章节可跳过 Tavily。
- 真实运行对比 Tavily 调用量、来源质量和 Reviewer 拒绝原因。

### [ ] 8.5 真实报告验证

任务：
- 使用同一主题运行新旧流程对比。
- 重点观察横向比较、证据边界、实施路径是否能引用前文证据。
- 检查是否减少“案例章节证据弱但仍按案例写”的拒绝。

验收标准：
- Reviewer 因“证据链断裂”“综合案例冒充实际案例”拒绝的频率下降。
- 横向比较和实施路径不再像独立搜索摘要，而是能承接前文案例。
- Tavily 调用量不明显增加，最好减少 synthesis 章节搜索。
- 全量单元测试通过。

## 风险

- 过度依赖前文材料可能导致后半部分信息不足，需要保留少量补搜能力。
- EvidenceMap 如果太长，会增加 LLM 成本并稀释重点。
- synthesis 章节跳过搜索后，可能缺少外部 benchmark，需要按章节类型谨慎启用。
- 阶段 8 会改变 Researcher 的核心控制流，应单独提交，避免和阶段 5 的搜索词优化混在一个提交中。
