# 项目 B：内容创作团队 (Content Creation Team)

## 场景描述
给定一个主题（例如：“探讨 2026 年 AI Agent 在企业数字转型中的实际应用案例”），由多个智能体协作完成一份深度调研报告。
协作链条：**产品经理** (规划大纲) → **研究员** (搜集资料) → **撰稿人** (起草全文) → **评审员** (质量审核与反馈)。

## 技术栈
- **LangGraph**：采用 Supervisor 模式进行多 Agent 总控调度。
- **Pydantic**：定义结构化的中间产物 (SOP Artifacts)，确保任务交接不丢信息。
- **Tavily**：研究员使用的联网搜索工具。
- **多 LLM Provider**: 通过统一工厂接入 DeepSeek / OpenAI / MiniMax / custom OpenAI-compatible API。
- **MemorySaver**: 断点续传与人类在环审核。

## 配置与运行
- 在项目根目录或本项目目录放置 `.env`，配置 `LLM_PROVIDER`、`LLM_MODEL`、`LLM_API_KEY` 和 `TAVILY_API_KEY`。也可以使用 provider 专属变量，例如 `DEEPSEEK_API_KEY` 或 `MINIMAX_API_KEY`。代码通过 `python-dotenv` 自动读取，不需要把密钥写入源码。
- 安装依赖：`pip install -r requirements.txt`
- 运行流程：`python supervisor_graph.py`
- 最终报告和引用清单会写入 `reports/` 目录。
- 日志级别可通过 `LOG_LEVEL` 控制，默认 `INFO`；排查细节时可设为 `DEBUG`。

## 坑位地图

| 编号 | 坑点 | 现象 | 根因 | 方案 | 状态 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **B1** | 角色越权 | 一个 Agent 越权干另一个的活 | 角色边界模糊 | 严格 Role 定义 + 强类型产物限制 | ✅ |
| **B2** | 信息丢失 | 上下游的结果下游看不全，导致输出断层 | 状态机 state 设计不合理 | 采用 `Annotated` 追加模式 + Pydantic 存储 | ✅ |
| **B3** | 评审死循环 | 评审员永远不满意，系统停不下来 | 缺少终止条件 | 引入 `review_count` 计数器 + 强制熔断 | ✅ |
| **B4** | 质量雪崩 | 某个角色输出质量差，下游全崩 | 缺少中间审核 | Researcher 输出执行程序化质量门禁，Reviewer 先做结构化引用校验 | ✅ |
| **B5** | 上下文膨胀 | 对话太长，Agent 反应变慢或出错 | 冗余信息堆积 | 曾使用摘要节点，现已从主流程移除以降低复杂度 | ⏸️ |
| **B6** | 版本混乱 | 不知道哪版是最终，修订历史丢失 | 缺乏状态快照 | 在 State 中记录 `draft_history` 版本列表 | ✅ |
| **B7** | 人类缺席 | 最终报告包含常识性错误 | 缺少断点机制 | 在大纲确认节点设置 `interrupt_before` | ✅ |
| **B9** | JSON 解析 | DeepSeek 在长文本下漏逗号或含引号 | `json_mode` 不稳 | 自研 `json_utils.py` 正则修复 + Pydantic 校验 | ✅ |
| **B10** | 引用缺失 | 报告无数据来源，可信度低 | Prompt 约束弱 | 强制数字编号引用 [1] + 参考资料列表 + 程序化校验 | ✅ |

## 最近改进 (Optimizations)
- **B7 人类在环**: 引入了 `MemorySaver` 持久化，现在 PM 规划大纲后会自动中断，等待用户在终端输入确认。
- **流程瘦身**: 主流程恢复为 `PM -> Researcher -> Writer -> Reviewer`，移除摘要节点，Reviewer 通过后也不再触发额外质量增强轮；Researcher 返工时完整重跑全章节，避免选择性补查和旧材料合并继续增加控制流复杂度。
- **Researcher 领域拆分**: 搜索规划、来源筛选、素材后处理和提示词契约已拆到 `content_research/`，`crew/researcher.py` 只保留角色编排与节点门禁。
- **B9/B10 解析与引用**: 解决了 DeepSeek 的 JSON 格式不稳定问题，并实现了严谨的引用溯源。
- **质量门禁**: 增加 `utils/quality.py`，对研究来源、来源排序、正文引用编号、章节编号、参考资料 URL 和字数做确定性校验。
- **写作风格**: 将质量要求调整为整篇报告层面的写作原则，避免各章节机械套用相同结构。

## 阶段性结论

当前先暂停继续优化。多轮实验说明：让 AI Agent 稳定写出高质量、可外发的深度报告非常困难。单次改动不一定带来改进，有时会降低报告质量；不断给流程加补丁会让代码越来越复杂，角色边界和控制流也会变得不可预测。

最近一次真实运行生成了 `reports/final_report_20260514_112128.md`，流程最终通过 Reviewer，结构、可读性和证据边界表达都有改善。但人工复核仍发现弱来源支撑核心数据、引用与句子支撑关系可能错配、综合案例缺少可追溯证据等问题。因此当前版本适合作为学习项目和流程实验样本，不应视为无需人工编辑即可外发的报告生产系统。

后续如重启，优先做离线评估和事实核查能力，而不是继续增加在线 Agent 分支。尤其需要先解决“引用是否真正支撑对应句子”的验证问题。

完整系统复盘见 `POSTMORTEM.md`。报告内容对 Agent 学习的启发见 `AGENT_REPORT_INSIGHTS.md`。通用方法论沉淀见 `../../docs/concepts/agent-design-methodology.md`。

`reports/final_report_20260512_144407.md` 仍是目前最好的阅读体验样本，但它是一次未完整保存生成条件的测试结果，不能当作稳定能力。后续如果重启，应先固定大纲、搜索材料、引用清单和 Writer 输入，验证能否复现类似质量；再逐步放开 PM 动态大纲和 Researcher 动态搜索。
