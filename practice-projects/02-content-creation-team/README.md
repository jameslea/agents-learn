# 项目 B：内容创作团队 (Content Creation Team)

## 场景描述
给定一个主题（例如：“探讨 2026 年 AI Agent 在企业数字转型中的实际应用案例”），由多个智能体协作完成一份深度调研报告。
协作链条：**产品经理** (规划大纲) → **研究员** (搜集资料) → **撰稿人** (起草全文) → **评审员** (质量审核与反馈)。

## 技术栈
- **LangGraph**：采用 Supervisor 模式进行多 Agent 总控调度。
- **Pydantic**：定义结构化的中间产物 (SOP Artifacts)，确保任务交接不丢信息。
- **Tavily**：研究员使用的联网搜索工具。
- **DeepSeek API**: 核心推理与生成引擎。
- **MemorySaver**: 断点续传与人类在环审核。

## 坑位地图

| 编号 | 坑点 | 现象 | 根因 | 方案 | 状态 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **B1** | 角色越权 | 一个 Agent 越权干另一个的活 | 角色边界模糊 | 严格 Role 定义 + 强类型产物限制 | ✅ |
| **B2** | 信息丢失 | 上下游的结果下游看不全，导致输出断层 | 状态机 state 设计不合理 | 采用 `Annotated` 追加模式 + Pydantic 存储 | ✅ |
| **B3** | 评审死循环 | 评审员永远不满意，系统停不下来 | 缺少终止条件 | 引入 `review_count` 计数器 + 强制熔断 | ✅ |
| **B4** | 质量雪崩 | 某个角色输出质量差，下游全崩 | 缺少中间审核 | 在 Researcher/Writer 节点加入自动校验逻辑 | ✅ |
| **B5** | 上下文膨胀 | 对话太长，Agent 反应变慢或出错 | 冗余信息堆积 | **摘要节点 (Summary Node)** 定期压缩历史 | ✅ |
| **B6** | 版本混乱 | 不知道哪版是最终，修订历史丢失 | 缺乏状态快照 | 在 State 中记录 `draft_history` 版本列表 | ✅ |
| **B7** | 人类缺席 | 最终报告包含常识性错误 | 缺少断点机制 | 在关键节点（大纲/发布）设置 `interrupt_before` | ✅ |
| **B9** | JSON 解析 | DeepSeek 在长文本下漏逗号或含引号 | `json_mode` 不稳 | 自研 `json_utils.py` 正则修复 + Pydantic 校验 | ✅ |
| **B10** | 引用缺失 | 报告无数据来源，可信度低 | Prompt 约束弱 | 强制内联引用 [来源: URL] + 生成来源汇总 | ✅ |

## 最近改进 (Optimizations)
- **B7 人类在环**: 引入了 `MemorySaver` 持久化，现在 PM 规划大纲后会自动中断，等待用户在终端输入确认。
- **B5 摘要助手**: 增加了 `summarizer` 节点，自动压缩冗长历史，确保 LLM 始终保持注意力。
- **B9/B10 解析与引用**: 解决了 DeepSeek 的 JSON 格式不稳定问题，并实现了严谨的引用溯源。
