> 评估日期：2026-05-15
> 目的：从项目 B 产出的高质量 Agent 主题报告中，提炼对后续 Agent 学习、项目 D-lite 代码执行安全与自愈最小实验、以及生产级 Agent 设计有参考价值的内容。

## 总体判断

这些报告最有价值的地方，不是具体数字，而是反复出现的一组共同判断：

1. Agent 的核心不是聊天能力，而是“感知、规划、工具调用、记忆、执行、反馈”的闭环。
2. 多 Agent 的价值来自职责、权限和上下文隔离，不来自角色命名。
3. 从 POC 到生产的主要障碍通常不是模型单点能力，而是数据、集成、权限、评估、审计和组织协作。
4. 高风险场景必须采用人机协作，而不是追求完全自动化。
5. 所有可行动的 Agent 都需要权限分级、行为日志、人工接管和 Kill-Switch。
6. ROI 要同时计算节省、收益、运维成本、人工复核成本和风险准备金。
7. 公开案例普遍证据不足，学习时必须区分“公开案例、厂商案例、综合案例、趋势观察”。

这些结论与我们在项目 B 中的体感一致：真正困难的是系统工程，而不是多写几个 Agent prompt。

## 对 Agent 学习路线的启发

### 1. 学 Agent 不能只学框架，要学系统边界

多份报告都把 Agent 描述为：

```text
大模型 + 工具调用 + 记忆/知识 + 执行环境 + 治理机制
```

这说明学习重点应从“会用 LangGraph/CrewAI/AutoGen”转向“如何设计一个受控系统”：

- 状态如何保存。
- 工具如何授权。
- 行为如何审计。
- 失败如何回滚。
- 输出如何评估。
- 人类如何介入。

框架只是表达这些边界的工具。

### 2. 先做低风险闭环，再扩大自主性

报告中反复出现的成熟路径是：

```text
建议模式 -> 人工审核执行 -> 半自动执行 -> 高置信度自动执行 -> 异常人工接管
```

这对我们的实践很重要。不要一开始就追求“全自动 Agent”。更合理的训练顺序是：

1. Agent 只读和分析。
2. Agent 生成建议，人类确认。
3. Agent 在沙箱内执行。
4. Agent 执行低风险动作。
5. Agent 执行高风险动作前强制审批。

项目 D 应从“沙箱内自愈”开始，而不是让 Agent 直接操作宿主机或真实服务。

### 3. 任务成功率比输出漂亮更重要

几份报告都提到评估指标问题，尤其是 `094020` 和 `094340`。对 Agent 项目来说，文章质量、回答流畅度只是表层指标，更重要的是：

- 任务完成率。
- 工具调用成功率。
- 人工介入比例。
- 错误修复成本。
- 失败可追溯性。
- 单次任务成本。
- 是否触发安全策略。

这也解释了项目 B 的问题：报告读起来不错，不代表引用支撑关系成立；Reviewer 通过，不代表事实可靠。

### 4. 证据边界本身就是 Agent 能力

`112128`、`094020`、`184444` 的一个进步是会主动区分公开案例、厂商案例、综合案例和趋势观察。

这对 Agent 开发非常关键：一个好 Agent 不应只给答案，还应能标注信心、证据等级和不可确定部分。

后续可以把这种思想迁移为通用 artifact 字段：

```text
claim
evidence
source_type
confidence
risk_level
requires_human_review
```

无论是报告生成、RAG 问答，还是代码修复，这种证据边界都能降低幻觉风险。

## 对项目 D-lite：代码执行安全与自愈最小实验的直接启发

这些启发已经同步补充到 `../../docs/practice-projects-plan.md` 的“项目 D-lite：代码执行安全与自愈最小实验”章节，作为项目 D-lite 开工前的设计约束。完整插件化自愈运维平台不再纳入当前实践范围。

### 1. 项目 D-lite 应定位为“最小受控自愈循环”

`152637` 中的智能运维章节很有参考价值：它把运维 Agent 的核心价值放在告警分析、根因定位、故障恢复和审计上，同时指出级联故障和不可追溯是最大风险。但作为当前学习项目，我们只吸收“自愈闭环、错误证据、验证和 trace”这些最小能力，不实现真实运维动作。

项目 D-lite 的主线应是：

```text
任务 -> 复制到临时目录 -> 生成/修改代码 -> 安全检查
    -> 执行 -> 错误分类 -> 修复 -> 再执行 -> 验证
```

不要把它做成“Coder 和 Reviewer 聊天直到看起来成功”。

### 2. 最小权限是关键设计原则

多份报告都强调最小权限、行为审计和人工确认。D-lite 不需要完整权限平台，但必须有简单硬边界：

| 边界 | 策略 |
|------|------|
| 写入范围 | 只允许写 challenge task 的临时副本 |
| 外部访问 | 默认禁止联网和安装依赖 |
| 命令执行 | 使用参数数组和超时，不执行任意 shell 拼接 |
| 危险 API | AST 检查拦截危险文件、网络和进程操作 |

这样比单纯在 prompt 里写“不要做危险操作”可靠得多。

### 3. 错误修复必须用测试验证

报告中的失败案例反复说明：Agent 的解释不等于正确。对项目 D-lite 来说，修复是否成功只能由客观信号判断：

- 命令退出码。
- 单元测试结果。
- 超时状态。
- 输出文件是否符合预期。
- 安全检查是否通过。

Reviewer 的语义判断只能作为补充，不能替代测试。

### 4. 轻量 Trace 应成为项目 D-lite 的核心能力

报告多次提到行为日志、审计和可追溯性。项目 D-lite 每一轮至少应记录：

- 输入任务。
- 生成代码或修复摘要。
- AST 安全检查结果。
- 执行命令。
- stdout/stderr 摘要。
- 错误分类。
- 修复计划。
- 重试次数。
- 最终状态。

否则一旦修复失败，就会重演项目 B 的问题：知道结果不好，但无法定位是哪一步导致的。

## 值得参考的具体内容

### `final_report_20260514_112128.md`

最值得参考的是它对 Agent 落地条件的总结：

- 数据基础薄弱会限制 Agent。
- 高风险行业需要私有化、权限、脱敏、日志。
- Agent 模式可分为全自动闭环、人机协同增强、多 Agent 协作网络。
- 规模化路线应包含评估、数据治理、试点、治理框架和扩展。

这份报告适合作为项目 D-lite 的“安全边界和路线图参考”，不作为扩展范围依据。

### `rejected_report_20260513_094340.md`

最有价值的是技术栈和指标框架：

- Agent = 大模型 + 工具调用 + 记忆/知识。
- 记忆可以分为短期工作记忆、长期用户记忆、外部知识记忆和运行态记忆。
- 工具调用可通过 Function Call、MCP 等协议对接外部系统。
- ROI 评估要区分过程指标和结果指标。

这份报告适合作为“Agent 系统组件清单”和“评估指标参考”。

### `final_report_20260513_152637.md`

最贴近项目 D-lite：

- 智能运维 Agent 可用于告警、诊断、恢复。
- 高风险变更必须保留人工审批。
- 审计日志要记录决策路径、工具调用和返回值。
- 渐进授权比一次性全自动更可靠。

这份报告适合作为项目 D-lite 的“自愈闭环参考”，真实运维场景只保留为未来方向。

### `rejected_report_20260514_094020.md`

最有价值的是证据边界意识：

- 大量 Agent 案例缺少企业名称、部署时间和第三方验证。
- 评估指标应包括任务成功率、修复成本、满意度和 ROI 净现值。
- 风险准备金应纳入 ROI。
- 没有权限审计的 Agent 部署风险极高。

这份报告适合作为“如何审慎阅读 Agent 行业资料”的样本。

### `final_report_20260512_144407.md`

最有价值的是写作结构，而不是事实强度：

- 背景和趋势先行。
- 多个场景案例形成节奏。
- 每个案例都有成功、挑战和横向/纵向分析。
- 后半部分收束到实施路径和高管建议。

这份报告适合作为“报告结构样本”，不适合作为事实依据。

## 需要警惕的内容

这些报告也暴露出几个问题：

1. 很多数字来自厂商报告、咨询报告或匿名综合案例，不能直接当作事实。
2. “2026 年预测”类内容容易混合趋势、愿景和实际部署。
3. 多 Agent、MCP、A2A、自治决策等概念容易被写成必然趋势，但真实落地还取决于业务和治理成熟度。
4. 部分报告把“可行模式”写成“实际案例”，这正是项目 B 后期反复修正的问题。
5. 对技术细节的描述多为架构层概括，不能替代代码级实践。

因此这些报告适合作为学习索引和问题地图，不适合作为可直接引用的事实库。

## 后续建议

1. 把 `152637`、`112128`、`094020` 三份作为项目 D-lite 前置阅读材料。
2. 从报告中抽取一份“Agent 生产化检查表”，覆盖数据、工具、权限、评估、trace、人工确认、退出机制。
3. 项目 D-lite 的 README 在开工前先写清楚：哪些动作只在临时目录或沙箱内执行，哪些指标证明自愈成功，哪些真实运维能力不做。
4. 如果未来重启项目 B，应把“证据边界字段”加入所有 ResearchMaterial 和 ReportClaim，而不是只靠 Writer 在正文中谨慎表达。
5. 保留这些高分报告作为“阅读样本”，但建立单独的事实核查层，避免把厂商宣传误当成可验证知识。

## 一句话结论

项目 B 的报告内容虽然不能直接作为可靠事实库，但已经清楚指出了 Agent 学习的主线：从模型能力转向系统能力，从角色协作转向受控工作流，从自动生成转向可验证、可审计、可恢复的工程闭环。
