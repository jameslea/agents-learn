## 一、文档定位

当前文档体系中已经有三类材料：

| 材料 | 价值 | 局限 |
|------|------|------|
| 权威资料和白皮书 | 抽象稳定，能解释 Agent 的核心组成 | 偏理论，不展开真实开发中的细碎问题 |
| 当前项目实践 | 问题真实，暴露了很多失败模式 | 细节繁杂，容易陷入具体脚本和局部实现 |
| 阶段综述文档 | 覆盖全面，能看到整体学习路线 | 不够聚焦每一层结构的工程阻力 |

因此，需要一篇中间层文档：

```text
用权威理论提供结构，
用项目实践补充失败模式，
用工程应对把经验沉淀成可复用判断。
```

本文和其他文档的关系是：

- `agent-core-structure.md`：解释 Agent 是什么，由哪些核心结构组成。
- `agent-theory-to-practice.md`：解释这些结构落地时会遇到什么问题。
- `agent-system-boundaries.md`：解释生产级 Agent 必须面对哪些系统边界。
- `agent-runtime-philosophy.md`：解释哪些能力应该沉淀为 Runtime。

## 二、核心判断

Agent 的高层结构可以简化为：

```text
Agent = Model + Tools + Orchestration
```

但真实开发中的问题通常不会以这么干净的形式出现。它们会表现为：

- 模型输出不稳定。
- 工具调用参数错误。
- 上下文遗漏关键约束。
- 状态无法恢复。
- 多 Agent 交接失败。
- 评估指标和真实质量不一致。
- 安全护栏误杀或漏判。
- Runtime 抽象过早，反而耦合具体业务。

所以更完整的工程判断是：

```text
理论结构用于定方向，
实践问题用于定边界，
失败模式用于指导工程设计。
```

一个 Agent 项目是否成熟，不只取决于它能否跑通 demo，还取决于它能否回答：

- 出错时能否定位原因？
- 中断后能否恢复？
- 工具是否可控？
- 输出是否可验证？
- 成本和延迟是否可管理？
- 失败是否能进入明确终态，而不是无限重试？

## 三、Model 层：模型能力与输出稳定性问题

模型是 Agent 的决策器，但模型输出不是传统函数返回。真实开发中，模型层常见问题包括：

| 问题 | 表现 | 工程应对 |
|------|------|----------|
| 输出不稳定 | 同一输入多次运行结果不同 | 降低自由度，使用结构化输出、示例、validator 和回归样本 |
| 不遵循 schema | JSON 缺字段、格式错误、类型不匹配 | 使用严格 schema、解析失败重试、输出修复器和单独验证步骤 |
| reasoning 不等于正确 | 模型解释看起来合理，但结论错误 | 用外部证据、测试、规则和人工复核验证关键结论 |
| provider 差异 | 不同模型对工具、格式、thinking 内容处理不同 | 引入 provider adapter，记录模型版本、延迟、成本和输出差异 |
| 小模型边界 | 小模型速度快但复杂任务容易失败 | 把小模型用于分类、抽取、局部 worker，把复杂判断交给强模型 |
| thinking 内容污染 | 思考内容进入最终答案或下游解析 | 在 provider 层清理输出，区分 reasoning、final 和 artifact |

模型层的核心应对原则是：

```text
不要把模型输出当成可信结果，
而要把它当成需要验证的候选决策。
```

## 四、Tools 层：工具接口、副作用与权限问题

工具让 Agent 能接触外部世界，也带来真实风险。模型错误回答可以重试，但错误工具调用可能会写文件、删数据、发消息、部署代码或泄露凭证。

常见问题包括：

| 问题 | 表现 | 工程应对 |
|------|------|----------|
| 工具描述不清 | 模型不知道什么时候该用哪个工具 | 写清用途、输入、输出、限制和反例 |
| 工具太多 | 模型选择困难，调用无关工具 | 分层工具集，按任务阶段暴露工具 |
| 参数错误 | 路径、ID、查询条件、格式不对 | schema 校验、参数规范化、调用前 dry-run |
| 副作用不可控 | 写入、删除、部署等动作被误触发 | 风险分级、人工审批、沙箱和最小权限 |
| 工具失败语义不清 | 模型不知道是否该重试 | 标准化错误码、错误摘要和可重试标记 |
| 环境差异 | 本地、CI、生产环境表现不同 | 显式环境探测、依赖检查和运行前预检 |

工具层的核心应对原则是：

```text
工具不是函数列表，
而是受权限、schema、副作用和观测约束的动作接口。
```

## 五、Orchestration 层：流程、循环与停止条件问题

编排层决定 Agent 如何从目标走向结果。理论上可以使用 workflow、状态机、ReAct、Planner-Executor、Reflection 或 Multi-Agent，但实践中最容易出问题的是控制流。

常见问题包括：

| 问题 | 表现 | 工程应对 |
|------|------|----------|
| 步骤过粗 | 一个 step 做太多事，失败后无法定位 | 拆分为可观察、可验证的 step |
| 步骤过细 | 流程复杂、调用次数多、成本高 | 合并低风险步骤，保留关键检查点 |
| 循环停止困难 | Agent 不断搜索、重试、反思 | 设置最大轮次、预算、完成条件和 blocked 终态 |
| retry 放大错误 | 同一错误被重复执行 | 记录失败原因，区分可重试和不可重试错误 |
| workflow 太死 | 稍有变化就无法处理 | 在固定流程中保留有限分支和人工介入点 |
| 自主循环太散 | 任务越做越偏 | 明确目标、约束、状态摘要和阶段性 review |

编排层的核心应对原则是：

```text
确定性部分用 workflow / state machine 承载，
开放判断交给模型，
高风险和低置信度场景停下来。
```

## 六、Context / Memory 层：信息选择与长期经验问题

上下文和记忆是 Agent 的核心难点之一。很多失败不是模型“不会”，而是模型没有看到正确的信息，或者看到了太多错误的信息。

上下文工程的完整讨论，包括上下文类型分层、write / select / compress / isolate / evaluate 生命周期、Context Builder 设计和安全治理，详见：[Agent 上下文工程：从 Prompt 拼接到可治理的工作视图](./agent-context-engineering.md)。

常见问题包括：

| 问题 | 表现 | 工程应对 |
|------|------|----------|
| 上下文膨胀 | 历史全部塞进 prompt | 用摘要、检索、artifact 引用和上下文窗口管理 |
| 关键约束丢失 | 模型忘记停止条件、格式要求或安全规则 | 把关键约束放入稳定系统上下文或 Runtime policy |
| 旧信息污染 | 已废弃结论继续影响后续步骤 | 给状态和 artifact 标记版本、有效性和来源 |
| 检索内容不可靠 | RAG 命中但证据不支持答案 | 引入证据表、引用约束和忠实度检查 |
| 记忆过期 | 历史经验不再适用 | 给记忆增加时间、范围、版本和验证状态 |
| 记忆与状态混淆 | 把当前任务进度当成长久记忆 | 明确区分 context、state、memory、artifact、trace |

上下文和记忆层的核心应对原则是：

```text
上下文是当前决策视图，
状态是当前任务进度，
记忆是跨任务经验，
三者不能混用。
```

## 七、State / Artifact / Trace 层：可恢复、可交接、可复盘问题

没有状态、产物和 trace 的 Agent 很难从 demo 走向可靠系统。它可能能跑完一次，但失败后无法解释，中断后无法恢复，多 Agent 之间无法稳定交接。

常见问题包括：

| 问题 | 表现 | 工程应对 |
|------|------|----------|
| 状态不可恢复 | 中断后只能重跑 | checkpoint 当前 step、预算、artifact 和失败原因 |
| 状态边界不清 | prompt、变量、文件、trace 混在一起 | 显式定义 RuntimeState、Artifact 和 Trace |
| 自由文本交接 | 下游 Agent 解析不稳定 | 使用 schema artifact、validator 和产物版本 |
| trace 记录太少 | 失败后不知道哪一步错 | 记录输入、输出、工具、错误、耗时和预算 |
| trace 记录太多 | 难以阅读，泄露敏感信息 | 分层 trace：原始记录、摘要、脱敏视图 |
| artifact 过度设计 | schema 很复杂但没人消费 | 只为真实下游和验证需求设计 artifact |

这一层的核心应对原则是：

```text
State 支撑恢复，
Artifact 支撑交接，
Trace 支撑复盘。
```

## 八、Evaluation / Guardrails 层：质量、安全与误判问题

评估和护栏是生产级 Agent 的基础能力，但它们本身也会出问题。LLM judge 不稳定，规则可能误杀，测试集可能给出虚假信心。

常见问题包括：

| 问题 | 表现 | 工程应对 |
|------|------|----------|
| LLM judge 不稳定 | 同一结果评分波动 | 固定 rubric，多次采样，结合规则和人工复核 |
| 指标和真实质量不一致 | 分数高但用户不满意 | 建立任务相关指标和真实样本回归集 |
| 测试集太小 | 小样本通过造成虚假信心 | 持续积累失败样本，做回归测试 |
| guardrail 误杀 | 正常任务被阻断 | 区分 hard block、soft warning 和 human review |
| guardrail 漏判 | 危险动作被放行 | 多层防护：输入、工具、输出、权限、沙箱 |
| 评估滞后 | 只在最终结果评估 | 在关键 step 加 evaluation hook |

这一层的核心应对原则是：

```text
评估不是给答案打分，
而是给 Agent 行为建立反馈和回归机制。
```

## 九、Multi-Agent 层：协作、交接与责任边界问题

多 Agent 是很容易被高估的方向。多个角色并不自动带来更高质量，反而可能放大错误、增加上下文污染和责任模糊。

常见问题包括：

| 问题 | 表现 | 工程应对 |
|------|------|----------|
| 角色名替代职责 | PM、Writer、Reviewer 名字很多，但行为相似 | 给每个 Agent 定义输入、输出、工具和评价标准 |
| 多角色放大错误 | 上游错误被下游继续采用 | 通过 evidence artifact 和 validator 阻断错误传递 |
| Reviewer 无约束力 | 只给泛泛意见 | 使用 rubric、评分项、必须修改项和 reject 条件 |
| 上下游责任不清 | 不知道谁引入了问题 | trace 每个 Agent 的输入、输出和产物 |
| 上下文互相污染 | 每个 Agent 看到过多无关信息 | 按职责裁剪上下文和工具权限 |
| 协作成本过高 | 多轮对话成本高、延迟长 | 只在确有并行、隔离或复核需求时使用多 Agent |

多 Agent 的核心应对原则是：

```text
多 Agent 的价值来自职责、工具、上下文和产物隔离，
不是来自角色名数量。
```

## 十、Runtime 层：公共能力抽象与具体场景适配问题

Runtime 是最有价值也最容易做重的方向。它的目标是为不同 Agent 提供公共运行能力，但如果抽象过早，就容易变成空壳、评测器或与某个具体项目强耦合的框架。

常见问题包括：

| 问题 | 表现 | 工程应对 |
|------|------|----------|
| 抽象过早 | Runtime 设计很大，但没有真实 Agent 使用 | 从一个真实场景抽取最小公共能力 |
| 抽象过窄 | 只服务某个项目，无法复用 | 区分 core、adapter、plugin 和 scenario |
| 变成评测器 | 只做结果评估，不参与执行 | Runtime 应覆盖 state、step、tool、artifact、trace |
| 状态边界不清 | RuntimeState、artifact、trace 混用 | 明确定义数据模型和生命周期 |
| 插件化复杂 | 通用工具接入成本高 | 先做少量稳定 adapter，再扩展插件协议 |
| 与运维环境绑定 | 依赖具体系统，无法通用 | 把环境相关能力放到 adapter，而不是 core |

Runtime 层的核心应对原则是：

```text
Runtime Core 管公共运行语义，
Adapter 管具体项目和环境差异，
Plugin / Skill 管可复用能力扩展。
```

## 十一、从当前项目得到的实践启发

当前项目的价值不在于每个实践都成功，而在于它们暴露了不同层面的真实问题。

| 项目 / 方向 | 暴露的问题 | 有价值的启发 |
|-------------|------------|--------------|
| 项目 A：RAG / 知识库问答 | 检索命中不等于答案忠实；语义相似不等于证据充分 | RAG Agent 的核心不是向量库，而是证据约束、引用边界和失败兜底 |
| 项目 B：内容团队多 Agent | 多角色协作效果不稳定，Reviewer 容易泛泛而谈 | 多 Agent 必须依赖结构化产物、rubric、事实核查和责任边界 |
| 项目 C：自主调研 | 长任务容易扩大范围，停止条件和成本控制困难 | 自主循环需要任务队列、预算、阶段性 review 和最终产物 schema |
| D-lite：代码自愈 | 自动修复可能引入回归，危险代码需要阻断 | 执行型 Agent 必须绑定安全策略、测试验证、最大尝试次数和 blocked 终态 |
| 项目 E：Agent Runtime | 早期方向曾偏向评测，后来转向 Runtime Core | Runtime 不是评测器，而是 state、step、tool、artifact、trace 的公共运行环境 |
| LLM provider 切换 | DeepSeek / MiniMax 在延迟、thinking 输出、配置方式上差异明显 | provider adapter、输出清理、耗时统计和模型配置隔离是基础能力 |
| 多模态研究 | 能力发展很快，但接入后会带来成本、安全和输入可信度问题 | 多模态更适合作为 Runtime 感知和表达扩展，而不是无边界直接接入 |

这些实践说明：失败或不稳定的项目不一定没有价值。它们往往比顺利 demo 更能暴露 Agent 系统真正需要沉淀的能力。

## 十二、阶段性结论

Agent 的理论结构很重要，因为它帮助我们避免陷入零散实现：

```text
Model + Tools + Orchestration
```

但工程实践同样重要，因为它告诉我们这些结构在哪里会坏：

```text
模型会不稳定，
工具会误用，
编排会失控，
上下文会污染，
状态会丢失，
产物会不可验证，
评估会误判，
Runtime 会抽象过早。
```

因此，后续继续做 Agent 实践时，不能只问：

```text
这个 Agent 能不能跑起来？
```

更应该问：

```text
它失败时能否定位？
它中断后能否恢复？
它的工具是否受控？
它的产物是否可验证？
它的行为是否可复盘？
它的经验是否能沉淀？
它的公共能力是否能复用？
```

这就是从理论到实践之间最重要的桥梁：把高层抽象转化为失败模式清单、工程边界和可复用运行时能力。
