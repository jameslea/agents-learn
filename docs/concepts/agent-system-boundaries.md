# Agent 系统开发边界：把不确定智能放进可治理的软件边界

> 更新时间：2026-05-19
>
> 本文整理 Agent Runtime 探索后的阶段性认知：开发生产级 Agent 并不是“传统软件接一个 LLM API”，而是在新的软件边界上做系统工程。当前还无法给 Agent Runtime 下最终定义，但已经能明确它必须面对哪些问题，以及为什么这些问题不能每次开发 Agent 时都重新临时处理。

## 一、核心判断

生产级 Agent 的难点，往往不在于让模型调用工具跑起来，而在于一整套非功能能力。

一个成熟 Agent 更接近：

```text
LLM 能力
  + 工具系统
  + 运行时
  + 治理系统
  + 评估系统
  + 运维系统
```

而不是：

```text
prompt + function calling
```

这意味着 Agent 开发已经进入新的软件边界：它既包含传统软件工程，也包含模型不确定性治理、工具副作用控制、评估回放、安全审批、成本延迟和上下文管理。

## 二、传统软件与 Agent 软件的根本差异

传统软件的核心假设通常是：

```text
输入确定 -> 程序逻辑确定 -> 输出可预测
```

Agent 软件的核心假设更接近：

```text
输入开放 -> 模型判断不稳定 -> 工具执行有副作用 -> 输出需要验证
```

这会导致系统复杂度系统性上升。

| 维度 | 传统软件 | Agent 软件 |
|------|----------|------------|
| 控制流 | 开发者显式写死 | 模型参与决策，控制流半开放 |
| 状态 | 数据库 / 内存状态 | 还包括上下文、记忆、trace、artifact、任务阶段 |
| 错误 | 异常、失败码、测试失败 | 幻觉、目标漂移、工具误用、上下文污染 |
| 权限 | 用户 / 服务 / API 权限 | 还要管理模型可调用工具、参数范围、读写边界 |
| 测试 | 单元、集成、端到端 | 还需要 eval、回放、人工复核、对抗样例 |
| 部署 | 服务稳定性和性能 | 还要管理模型版本、prompt 版本、成本延迟和随机性 |
| 安全 | 注入、越权、数据泄露 | 还多了 prompt injection、工具滥用、模型诱导执行 |
| 产物 | 函数返回 / 页面 / 文件 | schema artifact、证据链、解释、可验证输出 |

因此，很多 Agent demo 看起来很快，但走向生产会突然变重。demo 只展示“模型能做事”，生产要证明：

```text
它能在边界内持续、可控、可复盘地做事。
```

## 三、生产级 Agent 必须面对的非功能能力

下面这些能力并不直接等同于业务功能，但如果缺失，Agent 很难稳定进入真实场景。

| 能力 | 需要回答的问题 |
|------|----------------|
| 任务边界 | 目标、输入、输出、停止条件如何定义？ |
| 状态管理 | 长任务如何 checkpoint、resume，如何避免上下文膨胀？ |
| 工具治理 | 哪些工具可用、能读写哪里、谁来审批高风险动作？ |
| 产物结构化 | Agent 之间是否通过 artifact / schema / validator 交接，而不是自由文本？ |
| Trace 与复盘 | 失败后能否知道哪一步、哪个输入、哪个工具导致问题？ |
| 评估体系 | 是否有测试、规则、人工复核和回归样本，而不是只看模型回答？ |
| 人工介入 | 高风险、低置信度、越权动作是否能停下来？ |
| 成本延迟 | provider、model、token、latency、重试、缓存是否可管理？ |
| 安全边界 | 是否防护提示注入、越权读写、危险命令和凭证泄露？ |
| 版本治理 | prompt、工具、策略、模型、数据是否有版本和回滚机制？ |
| 部署运维 | 队列、并发、隔离、监控、告警、审计、权限系统如何落地？ |

这些能力就是 Agent Runtime、Governance、Observability 和 Evaluation 逐渐出现的原因。它们不是装饰性模块，而是把 Agent 从 demo 推向真实系统的边界条件。

### 3.1 任务边界

Agent 的任务边界如果不清楚，系统很容易从“执行任务”滑向“自由发挥”。传统软件通常由 API、表单、流程和数据库约束输入输出；Agent 则常常面对开放目标，例如“帮我研究一下”“自动修复问题”“生成一份报告”。这类目标如果不被拆成明确的输入、产物、成功标准和停止条件，模型会不断扩展任务范围，甚至把模糊目标解释成不必要的复杂行动。

任务边界要回答的不只是“用户想要什么”，还包括“哪些事情不做”。例如报告生成 Agent 是否需要联网、是否需要事实核查、是否需要引用、什么时候可以停止、输出必须是什么格式，这些都应该进入任务契约。没有任务契约，后面的工具治理、状态管理、评估和人工介入都会缺少依据。

更具体地说，一个 Agent 任务至少应明确下面这些边界：

| 边界项 | 需要明确的内容 |
|--------|----------------|
| 目标边界 | 本次任务到底交付什么，不交付什么 |
| 输入边界 | Agent 可以使用哪些用户输入、文件、知识库、网页或 API |
| 输出边界 | 最终产物是回答、报告、代码 patch、配置变更，还是结构化 artifact |
| 行动边界 | Agent 可以只分析、可以写文件、可以调用外部服务，还是可以执行命令 |
| 质量边界 | 什么算完成，什么算不合格，是否需要引用、测试或人工复核 |
| 停止边界 | 到达什么条件必须停止，而不是继续搜索、继续修复或继续扩写 |
| 风险边界 | 哪些动作必须人工审批，哪些动作永远不允许 |

比如“生成一份调研报告”不是一个足够清晰的任务边界。更清晰的表达应接近：“基于指定资料和最多 5 个公开网页，生成一份 Markdown 报告；必须包含引用、风险边界和结论；不得修改本地文件以外的资源；如果证据不足，输出待确认问题而不是继续扩展范围。”这样的边界能让 Runtime 判断哪些工具可用、哪些输出可接受、什么时候需要停止。

任务边界模糊时，常见问题包括：Agent 为了补足信息不断扩大搜索范围；把“建议修改”误解成“直接修改”；把“总结已有资料”扩展成“重新做研究”；或者在输出不合格时不知道是继续重试、请求人工确认，还是返回失败。任务边界的作用，就是把这些选择提前显式化。

### 3.2 状态管理

Agent 的状态不只是程序变量或数据库记录。它还包括当前任务阶段、上下文摘要、模型已经看过的证据、已经生成的 artifact、工具调用结果、失败原因和下一步计划。长任务如果只依赖对话上下文，很快会遇到上下文膨胀、旧信息污染、关键证据丢失和无法恢复的问题。

因此，生产级 Agent 需要 checkpoint 和 resume。checkpoint 不只是保存“文本历史”，而是保存结构化状态：当前 step、已完成 step、产物引用、预算消耗、错误摘要和人工审核状态。这样任务中断后才能恢复，失败后才能定位，版本升级后才能判断旧状态是否还能继续使用。

Agent 状态可以粗略分为几类：

| 状态类型 | 示例 | 作用 |
|----------|------|------|
| 任务状态 | task_id、目标、当前阶段、最终状态 | 说明任务执行到哪里 |
| 步骤状态 | step_id、started、passed、failed、skipped | 支撑 step 级复盘和 resume |
| 上下文状态 | 当前摘要、保留证据、被丢弃的信息 | 控制上下文膨胀和信息污染 |
| 产物状态 | artifact id、文件路径、schema、版本 | 让中间结果可引用、可校验 |
| 工具状态 | 工具调用次数、输入摘要、返回结果、错误 | 判断是否超限、是否可重试 |
| 预算状态 | token、时间、费用、重试次数 | 防止任务无限扩大 |
| 审批状态 | 待审批动作、审批结果、审批人、理由 | 支撑高风险动作暂停和恢复 |

这里最容易混淆的是“上下文”和“状态”。上下文是模型当前看到的文本窗口，适合放短期工作记忆；状态是 Runtime 可持久化、可校验、可恢复的事实记录。一个重要原则是：不能把关键事实只放在上下文里。凡是后续步骤必须依赖、失败后需要复盘、恢复时必须重建的信息，都应该进入 state 或 artifact。

状态管理还要处理“恢复后是否还能继续”的问题。比如一个代码修复 Agent 在第三步失败，resume 时不应该重新执行已经通过的安全检查，也不应该丢失前两步生成的错误摘要和 patch 草稿。它应该知道哪些 step 已完成，哪些 artifact 可复用，哪个工具调用失败，失败是否可重试，当前模型或工具版本是否和 checkpoint 兼容。

状态管理做不好时，Agent 会出现几类典型症状：每次失败都只能从头开始；长任务越跑越偏但不知道从哪一步偏离；模型重复调用同一个工具；重要证据被上下文压缩丢掉；人工批准后无法从暂停点继续。这些问题表面上像 prompt 问题，本质上往往是 Runtime 没有管理好状态。

### 3.3 工具治理

工具让 Agent 从“回答问题”变成“改变环境”。一旦 Agent 能读文件、写文件、调用 API、执行命令或操作浏览器，工具治理就变成核心问题。模型不应该天然拥有所有工具，也不应该自由决定工具参数。否则，提示注入、误判、越权读取、错误写入和高风险副作用都会变成真实系统风险。

工具治理至少包括四层：工具是否允许调用、参数是否在范围内、调用风险等级是否需要审批、调用次数和超时是否受限制。更进一步，还要区分读权限、写权限、网络权限、凭证权限和高风险操作权限。一个成熟 Runtime 不应只暴露函数列表，而应把工具作为受策略约束的能力。

工具治理可以从下面几个维度检查：

| 治理维度 | 典型问题 |
|----------|----------|
| 工具清单 | 当前任务是否真的允许调用这个工具？ |
| 参数边界 | 文件路径、URL、命令参数、查询条件是否在允许范围内？ |
| 权限类型 | 这是读操作、写操作、网络访问、命令执行，还是外部系统变更？ |
| 风险等级 | 是否会修改状态、产生费用、影响用户数据或触发不可逆副作用？ |
| 调用预算 | 是否超过最大调用次数、最大耗时或最大重试次数？ |
| 审批策略 | 高风险动作是自动执行、阻塞，还是进入人工确认？ |

工具治理做不好时，Agent 的错误会从“回答错了”升级为“环境被改错了”。例如模型误把临时目录当成项目目录、把查看命令替换成删除命令、把草稿建议直接写入生产配置，或者被网页中的 prompt injection 诱导去读取凭证文件。这些风险不能只靠提示词约束，必须在工具执行前由策略层拦截。

### 3.4 产物结构化

很多 Agent demo 依赖自然语言在步骤之间传递信息，例如“上一步总结如下”“请基于上面的内容继续”。这种方式适合演示，但不适合长期运行。自然语言交接很难校验、很难 diff、很难复用，也很难在失败后定位哪个字段出错。

结构化产物的意义是把关键中间结果变成可校验对象。RAG 的答案应包含引用和证据，代码修复应包含目标文件、修复摘要和验证结果，报告评审应包含问题列表、严重程度和改进计划。Artifact、schema 和 validator 能让 Agent 的输出从“看起来合理”变成“可以被系统检查和后续消费”。

可以把 artifact 理解为 Agent 系统里的“可审计交接单”。它不要求所有内容都结构化，但关键字段必须明确。例如：

| 场景 | 关键 artifact |
|------|---------------|
| RAG 问答 | answer、citations、retrieved_chunks、confidence、unsupported_claims |
| 调研报告 | outline、source_notes、evidence_table、risk_notes、final_report |
| 代码修复 | error_summary、patch、changed_files、test_result、regression_risk |
| 内容评审 | issue_id、severity、evidence、recommendation、approval_status |

结构化产物还有一个重要作用：把上下文从“长文本堆叠”变成“引用稳定对象”。后续步骤不必重新阅读全部历史，只需要读取必要 artifact。这样既减少上下文压力，也让评估、回放和人工审核更清楚。

### 3.5 Trace 与复盘

Agent 失败时，单看最终回答通常没有价值。真正需要知道的是：任务从哪里开始，经历了哪些 step，调用了哪些工具，输入是什么，工具返回了什么，模型基于什么证据做了判断，在哪一步偏离目标或触发安全边界。

Trace 是 Agent 系统的黑匣子记录。它不只是日志，而是可回放的执行证据。好的 trace 应该能回答：为什么这次通过、为什么这次失败、失败能否重试、是否需要人工介入、是否是模型问题、工具问题、数据问题还是策略问题。没有 trace，Agent 的调试会退化成反复猜 prompt。

一个可用 trace 至少应覆盖：

| Trace 内容 | 用途 |
|------------|------|
| task_started / task_finished | 明确任务边界和最终状态 |
| step_started / step_finished / step_failed | 定位流程停在哪一步 |
| tool_decision / tool_called | 区分工具被允许、阻塞、审批或执行 |
| artifact_created | 追踪关键中间产物和最终产物 |
| validation / evaluation | 记录判断通过或失败的依据 |
| human_review | 记录人工介入请求和审批结论 |

Trace 也要控制粒度。过粗的 trace 只能看到“失败了”；过细的 trace 会泄露敏感信息或变成无法阅读的日志噪声。比较合理的方式是记录输入摘要、产物引用和关键指标，把大段正文、文件内容和模型输出放到 artifact store 或受控日志中。

### 3.6 评估体系

传统软件可以大量依靠单元测试和集成测试；Agent 软件还需要评估体系。因为 Agent 输出不是固定字符串，模型行为也会随 provider、model、prompt、上下文和温度变化。只看一次回答是否“像是对的”，无法支撑回归和上线。

评估体系应该分层：确定性规则负责格式、结构、引用、权限和硬约束；任务测试负责客观结果；人工复核负责高风险和主观质量；LLM judge 只能作为辅助信号，不能替代所有裁决。评估的目标不是给 Agent 打一个漂亮分数，而是让系统能持续发现退化、定位风险并决定是否可以交付。

更具体地说，评估不应只有一个总分，而应拆成多个信号：

| 评估信号 | 示例 |
|----------|------|
| 格式正确性 | JSON schema、Markdown 结构、必填字段 |
| 事实忠实度 | 回答是否被引用支撑，是否出现 unsupported claim |
| 任务完成度 | 是否交付了要求的 artifact，是否满足成功标准 |
| 安全合规 | 是否调用越权工具，是否触发敏感数据泄露 |
| 回归稳定性 | 同一批 case 在模型或 prompt 变化后是否退化 |
| 人工质量 | 专家对可读性、业务适配性和风险边界的判断 |

LLM judge 可以帮助发现表达质量、逻辑漏洞和主观问题，但它本身也是不稳定模型调用。因此，生产级评估通常需要“确定性规则 + 任务测试 + 人工抽检 + LLM 辅助”的组合，而不是把最终裁决完全交给另一个模型。

### 3.7 人工介入

生产级 Agent 不应该假设所有流程都能自动完成。高风险动作、低置信度判断、越权请求、事实不确定、成本异常和用户意图模糊，都应该有机会停下来。人工介入不是失败，而是运行时治理的一部分。

关键在于人工介入不能只是临时 `input()` 或聊天确认。它应该成为状态机中的明确状态，例如 `needs_human`，并记录请求原因、待审批动作、输入摘要、风险等级、审批人、审批结论和时间。这样人工判断才能进入 trace 和审计链路，后续也能复盘为什么当时允许或拒绝。

常见的人工介入触发点包括：

| 触发点 | 例子 |
|--------|------|
| 高风险工具 | 写配置、执行命令、删除文件、调用付费 API |
| 低置信度 | 证据不足、模型自评不确定、检索结果冲突 |
| 越权倾向 | 请求访问未授权目录、外部系统或用户数据 |
| 成本异常 | 重试过多、token 消耗过大、任务耗时异常 |
| 业务判断 | 法务、财务、医疗、人事等需要责任主体的判断 |

人工介入的目标不是让人替 Agent 做所有事，而是在关键边界上建立责任转移点。系统应该清楚告诉人：为什么停下、准备做什么、风险是什么、批准后会发生什么、拒绝后如何收束。

### 3.8 成本延迟

Agent 往往不是一次模型调用，而是多次模型调用、检索、工具执行、重试和评估组成的链路。成本和延迟会在这个链路中被放大。一个单次回答便宜的模型，在多轮工具调用和失败重试后，可能变得昂贵且不可预测。

因此，生产级 Agent 需要记录 provider、model、token、latency、失败原因、重试次数和缓存命中情况。成本延迟控制不是单纯省钱，而是保证系统行为可预期：什么时候用强模型，什么时候用小模型，什么时候缓存，什么时候停止重试，什么时候把任务转给人工。

成本延迟治理可以从三个层面考虑：

| 层面 | 关注点 |
|------|--------|
| 单次调用 | provider、model、token、latency、错误率 |
| 单个任务 | 总调用次数、总耗时、总成本、重试次数 |
| 系统层面 | 并发、队列积压、缓存命中率、限流和降级 |

很多 Agent 的成本问题来自“隐形循环”：模型不断搜索、不断反思、不断重写，但没有明确收益。Runtime 需要预算和停止条件，例如最大工具调用数、最大搜索轮次、最大重试次数、最大 token 成本。当达到预算上限时，系统应该输出当前证据和未完成原因，而不是继续消耗。

### 3.9 安全边界

Agent 系统的安全问题比传统软件更复杂，因为模型会阅读不可信内容，并可能把这些内容转化为工具调用。网页、文档、邮件、issue、日志都可能包含 prompt injection，诱导模型泄露数据、调用危险工具或忽略原有规则。

安全边界不能只写在 prompt 里。它需要落实到 Runtime 和工具层：路径作用域、网络开关、危险命令拦截、凭证隔离、参数校验、高风险审批、输出脱敏和审计记录。模型可以建议动作，但是否允许执行，必须由策略系统裁决。

安全边界需要特别关注“不可信输入”。任何来自网页、文档、邮件、用户上传文件、日志、issue 的内容，都不应直接拥有改变系统行为的权限。模型可以阅读这些内容，但不能因为内容中写着“忽略之前的规则”就改变工具策略。

一个基本原则是：安全规则必须在模型外部生效。Prompt 可以提醒模型遵守规则，但真正的执行权限要由 Runtime、policy、sandbox 和权限系统控制。这样即使模型被诱导，危险动作也会在工具层被拦截。

### 3.10 版本治理

Agent 系统的行为由多种可变因素共同决定：模型版本、prompt、工具实现、工具策略、检索数据、评估规则、上下文压缩方式和外部 API。任何一项变化都可能改变最终行为。如果没有版本治理，线上问题很难复现，也很难判断是哪次变更导致退化。

版本治理需要把 Agent 当成完整系统看待，而不是只记录代码 commit。一次运行最好能知道使用了哪个模型、哪版 prompt、哪组工具、哪份策略、哪些数据和哪些评估规则。这样 trace、manifest、artifact 和评估结果才能真正支持回归分析和灰度发布。

Agent 版本至少包含：

| 版本对象 | 说明 |
|----------|------|
| 代码版本 | Runtime、adapter、tool handler 的代码 commit |
| Prompt 版本 | system prompt、任务 prompt、工具说明、评审标准 |
| 模型版本 | provider、model、参数、是否启用 thinking/json mode |
| 策略版本 | 工具权限、路径范围、审批规则、预算限制 |
| 数据版本 | 知识库、索引、检索配置、样本集 |
| 评估版本 | eval case、评分规则、阈值、人工标注版本 |

没有版本治理时，Agent 问题会变得很难讨论：同一句“昨天还能跑，今天不行了”，可能来自模型升级、prompt 修改、数据变化、工具返回格式变化或评估阈值变化。版本信息进入 manifest 和 trace，是后续回归分析的基础。

### 3.11 部署运维

Agent 一旦进入生产，就会面对传统后端同样的问题：队列、并发、隔离、超时、重试、监控、告警、审计和权限。但 Agent 还多了模型不稳定、工具副作用、长任务状态和成本控制。部署运维不是最后才考虑的包装层，而是影响 Runtime 设计的基础约束。

例如长任务需要队列和 checkpoint；并发执行需要 run id、锁和隔离目录；工具调用需要超时和限流；高风险动作需要审批记录；模型调用需要失败降级和告警。没有这些能力，Agent 在 demo 中可以运行，在生产中却很难稳定承担责任。

部署运维还要决定 Agent 以什么形态运行：同步 API、异步任务、定时任务、人工审批流，还是事件驱动 worker。不同形态对 Runtime 的要求不同。同步 API 更关注超时和快速失败；异步任务更关注 checkpoint、队列和重试；审批流更关注暂停、恢复和审计；事件驱动系统更关注幂等和重复消息处理。

因此，生产级 Agent 的部署不是“把脚本放到服务器上”。它需要明确运行模式、隔离边界、失败策略和观测指标。否则系统一旦出现慢响应、重复执行、半途失败或工具副作用，就很难判断是模型问题、业务问题还是运维问题。

## 四、Agent Runtime 探索带来的认知

这次项目 E / Agent Runtime & Governance Lab 的探索，并没有得出一个最终框架，也不应该过早声称已经形成产品级 Runtime。

更合理的定位是：

```text
Agent Runtime 探索性研究 + 最小工程验证
```

它的阶段性价值在于：

- 从 A/B/C/D-lite 的实践中抽出了 Runtime 的必要能力。
- 纠正了“观测/评测”和“Runtime execution”的边界。
- 做出了一个可运行的 Runtime Core MVP。
- 用 B-runtime-lite 和 D-lite 验证了 adapter、state、artifact、trace、tool policy、run_id、manifest 等概念。
- 暴露了复杂度边界：再往后会进入策略系统、调度系统、并发系统、部署系统和观测平台。

因此，当前不能给 Agent Runtime 下最终定义，但已经知道它至少不是简单 Agent demo。它更像一组围绕任务执行、状态、工具、产物、trace、安全和恢复的公共运行支撑能力。

## 五、一个重要设计分界

后续再开发具体 Agent 项目时，应优先问一个问题：

```text
这个能力属于业务逻辑，还是属于 Runtime 公共能力？
```

如果属于业务逻辑，它应该留在具体 Agent 项目中，例如选题、写作策略、研究方法、行业规则、业务审批流程。

如果属于 Runtime 公共能力，它就不应该每次重新写一遍，例如：

- 状态持久化。
- 上下文治理。
- 工具权限。
- 安全边界。
- 结构化产物。
- 可回放 trace。
- 评估体系。
- 人工介入。
- 成本和延迟控制。
- 部署与版本治理。

能分清这一点，比继续堆更多 demo 更重要。因为它决定了系统复杂度是被公共框架吸收，还是在每个 Agent 项目里重复失控。

## 六、当前阶段的结论

Agent 的核心不是自主性越强越好，而是把不确定的智能能力放进可治理的软件边界里。

这个边界包括：

```text
任务边界
状态边界
工具边界
权限边界
产物边界
评估边界
人工介入边界
成本延迟边界
部署运维边界
```

这也是 Agent Runtime 探索真正想摸清楚的东西。

当前最务实的态度是：

```text
不要急于抽象出最终框架。
先通过具体项目暴露边界，再把确实重复出现的能力沉淀为 Runtime 公共能力。
```

这样既承认 Agent 系统开发的复杂性，也避免过早把学习项目推向产品级框架工程。
