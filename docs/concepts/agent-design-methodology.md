# Agent 设计方法论：从角色实验到可控系统

> 本文承接 `practice-projects/02-content-creation-team` 的阶段性经验：多 Agent 的关键不是角色数量，而是边界、状态、工具、评估和可回放性。

## 核心结论

当前阶段应把 Agent 看成“受控工作流中的动态判断节点”，而不是默认把复杂任务拆成多个会说话的角色。

一个可维护的 Agent 系统，优先级应是：

1. 先定义任务是否真的需要 Agent。
2. 再设计 workflow、state、artifact 和 eval。
3. 最后才决定是否需要多 Agent、handoff 或自主循环。

如果一个任务可以用固定流程、普通函数、检索、规则或单次 LLM 调用解决，就不应提前引入复杂多 Agent。

## 工作流优先，而不是角色优先

很多失败的多 Agent 原型，本质是“多个 prompt 串联”。角色名看起来清晰，但系统边界并没有变清晰。

判断一个角色是否有资格成为独立 Agent，应看四件事：

| 问题 | 合格标准 |
|------|----------|
| 它是否拥有独立目标？ | 目标能独立评估，不依赖模糊审美判断 |
| 它是否需要独立工具？ | 工具权限和失败后果不同于其他节点 |
| 它是否需要独立上下文？ | 给它全局上下文会明显增加干扰或成本 |
| 它是否能产出稳定 artifact？ | 输出可以被结构化校验、版本化、回放 |

如果答案多数是否，它更适合作为 workflow 中的一个 LLM 节点，而不是独立 Agent。

## Agent 五层设计法

### 1. 场景层：定义成功条件

先写清楚用户任务、失败成本和验收方式。

好的场景定义：

- 有明确输入和输出。
- 有可观察的完成标准。
- 有失败后的降级策略。
- 知道哪些步骤必须人类确认。

不好的场景定义：

- “生成一篇高质量报告”。
- “像专家一样调研”。
- “自动解决所有问题”。

这类目标太宽，必须拆成可验证子目标，例如“引用是否支撑句子”“是否完成 8 个调研子问题”“是否通过 3 个沙箱任务”。

### 2. 状态层：让系统可恢复

Agent 状态不应只是对话历史。生产级状态至少包含：

- 原始目标。
- 当前阶段。
- 已完成 artifact。
- 工具调用记录。
- 失败原因。
- 重试次数。
- 成本预算。
- 人类确认点。

原则是：任意一次失败后，都能回答“停在哪、为什么停、能否继续、继续会不会重复副作用”。

### 3. 产物层：用 artifact 代替长对话

Agent 之间的交接应依赖结构化产物，而不是把上一轮完整对话塞给下一轮。

推荐模式：

```text
Input Contract -> Agent/Node -> Artifact -> Validator -> Next Node
```

Artifact 应具备：

- Pydantic 或等价 schema。
- 版本号或生成时间。
- 来源或证据引用。
- 校验状态。
- 可被离线评估脚本读取。

项目 B 的经验说明：只记录“最终报告”不够。高质量样本若没有保存大纲、搜索材料、引用清单、prompt 和模型参数，就不能复现，也不能沉淀为系统能力。

### 4. 工具层：权限小于能力

工具设计比 prompt 更重要。一个 Agent 能调用什么工具，决定了它能造成什么后果。

工具应按风险分级：

| 等级 | 示例 | 策略 |
|------|------|------|
| 只读 | 搜索、读文件、查数据库 | 默认允许，记录 trace |
| 写内部 | 写草稿、生成报告、更新本地缓存 | 允许，但需要版本化 |
| 写外部 | 发邮件、提交 PR、删数据、执行命令 | 强制确认或沙箱 |

工具描述要像 API 文档，而不是像自然语言提示词。必须写清：

- 参数含义。
- 输入格式。
- 输出格式。
- 失败类型。
- 禁止行为。
- 示例调用。

### 5. 评估层：先锁基线，再优化

不要在没有基线的情况下调 prompt。每次改动都应能回答：

- 哪些样本变好了？
- 哪些样本变差了？
- 是否只是过拟合一个失败样本？
- 成本和延迟是否增加？
- 人类复核是否仍发现同类问题？

评估至少分三类：

| 类型 | 适用场景 | 示例 |
|------|----------|------|
| 确定性评估 | 格式、结构、引用编号、工具调用结果 | 单元测试、正则/schema 校验 |
| LLM-as-a-Judge | 语义质量、相关性、完整度 | 大纲评分、报告质量评分 |
| 人类复核 | 事实支撑、业务风险、最终可用性 | 引用是否真的支撑句子 |

项目 B 的关键教训是：Reviewer 通过不等于报告质量稳定。尤其是“引用是否真正支撑对应句子”这种问题，必须独立设计核查能力。

## 什么时候使用哪种模式

| 模式 | 适合 | 不适合 |
|------|------|--------|
| 单次 LLM 调用 | 低风险、短任务、可直接判断输出 | 需要工具和多步状态的任务 |
| Prompt chaining | 可分解成固定步骤的任务 | 步骤数量或顺序高度不确定的任务 |
| Routing | 输入类型清晰、需要不同处理路径 | 分类边界模糊或样本太少 |
| Evaluator-optimizer | 有清晰评估标准，反馈能稳定改进输出 | 质量标准主观且不可测 |
| Orchestrator-workers | 子任务数量不确定，但结果可合成 | worker 输出无法独立评估 |
| Autonomous loop | 开放目标、需要动态计划和环境反馈 | 高风险外部动作、无预算限制 |
| Multi-agent handoff | 需要状态切换、上下文隔离或权限隔离 | 只是为了模拟组织角色 |

## 多 Agent 的判断清单

引入多 Agent 前，逐项检查：

- 是否存在一个单 Agent 无法承载的上下文或工具集合？
- 是否需要并行执行？
- 是否需要把不同权限隔离开？
- 每个 Agent 的输出是否能被独立校验？
- handoff 是否有明确触发条件？
- 是否有最大轮数、预算和失败终止条件？
- trace 能否还原每个 Agent 的输入、输出和工具调用？

如果这些问题答不清，多 Agent 通常会增加复杂度，而不是增加能力。

## 面向项目 D-lite 的设计提醒

项目 D-lite 是代码执行安全与自愈最小实验，风险比项目 B 更高，因为它涉及代码生成和执行。它不再承担通用运维平台、插件化架构、Kubernetes/SSH/CI 适配等目标。

进入项目 D-lite 前，应把设计重点放在：

- 临时目录或沙箱隔离优先于 prompt 禁止。
- AST 安全检查优先于事后审查。
- 错误分类优先于直接把完整 traceback 塞回模型。
- 最大重试次数和超时是系统能力的一部分。
- 修复成功必须由测试或命令结果验证。
- 每次实验必须只操作 challenge task 的副本。
- 复杂运维插件、生产动作、人工审批流都放到未来方向。

项目 D-lite 不应只是 AutoGen 多角色聊天 demo，而应是一个“最小受控自愈循环”：

```text
Task -> Prepare Workspace -> Coder -> AST Check -> Execute
     -> Error Classifier -> Repair -> Verify -> Final Report
```

## 推荐实践顺序

1. 对项目 B 做复盘，提取失败模式。
2. 把失败模式转成项目 D-lite 的安全边界和验证约束。
3. 先实现 AST 检查、临时目录执行、错误分类、终止条件和 trace。
4. 再接入单 Agent 修复。
5. 多 Agent、插件化运维和真实环境适配只作为未来方向记录。
