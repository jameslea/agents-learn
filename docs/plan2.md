# Agent 第二阶段学习路线图：从框架分类到模式体系

> 第一阶段已经完成了 Agent 的核心概念、LangGraph 状态机、多 Agent 协作、Self-RAG，以及 smolagents / CrewAI / AutoGen 的横向入门。
> 第二阶段的目标，是从“会用几个框架”升级为“理解 Agent 领域的主要架构模式，并知道不同框架解决的是哪一类问题”。

---

## 一、为什么需要第二阶段

当前仓库已经覆盖了 Agent 开发中最重要的一条主线：

```text
Tool / Prompt / Memory
  -> ReAct
  -> LangGraph 状态机
  -> Supervisor 多 Agent
  -> Self-RAG
  -> 多框架横向对比
```

这条路线解决的是“如何让 LLM 可靠地调用工具、管理状态、完成多步骤任务”。但 Agent 领域并不只有 LangGraph、CrewAI、AutoGen 和 smolagents。

继续深入时，需要补上六类能力：

- **数据中心型 Agent**：让 Agent 更好地理解、索引和检索企业私有数据。
- **SOP 驱动型多 Agent**：把软件工程、内容生产等标准流程拆成稳定产物。
- **自主任务循环**：从一个长期目标出发，自动生成任务、执行任务、反思结果并更新计划。
- **技能库与长期学习**：把一次成功经验沉淀成可复用技能，而不是每次从零推理。
- **低代码 Agent 平台**：理解 Dify、Coze 这类平台如何把 Agent 工作流产品化。
- **生产级 Agent 工程**：评估、观测、成本、权限、失败恢复和人类介入。

第二阶段不是简单增加更多 Hello World，而是围绕这些模式建立更完整的 Agent 知识地图。

---

## 二、Agent 框架的四大类型

Agent 框架可以按照功能侧重点、架构设计理念和应用场景分为四大类：

1. 多智能体协同框架
2. 自主探索型 / 通用任务框架
3. 底层编排与工具集成框架
4. 低代码 / 可视化 Agent 平台

这四类框架没有绝对高下，它们解决的问题不同。学习时最重要的不是记住名字，而是理解每一类框架背后的核心隐喻。

---

## 三、多智能体协同框架

### 核心隐喻

多智能体协同框架的核心是“分工”。它们把复杂任务拆给多个角色：研究员、分析师、程序员、测试员、主管等。每个 Agent 有不同职责，通过对话、消息传递或结构化产物协作完成任务。

### 主要特点

- 强调角色分工，而不是单个 Agent 全能解决所有问题。
- 常见结构是 Supervisor、Group Chat、Team、Crew、Company。
- 适合需要多轮讨论、审核、修正和产物交接的复杂任务。
- 风险在于对话可能发散，因此需要明确终止条件、角色边界和流程约束。

### 代表框架

| 框架 | 主要特点 | 适用场景 |
| :--- | :--- | :--- |
| **AutoGen** | 以对话流为核心，支持多 Agent 群聊、代码执行、人类介入。 | 软件开发、数据分析、自动化运维、自愈式任务。 |
| **CrewAI** | 以角色、目标、Backstory 和 Task 为中心，强调类似职场协作的 SOP。 | 内容生产、研究报告、流程清晰的企业任务。 |
| **MetaGPT** | 将软件工程 SOP 显式引入 Agent 协作，强调 PRD、设计文档、架构图等标准产物。 | 软件工程自动化、需求分析、结构化项目交付。 |
| **ChatDev** | 模拟一个虚拟软件公司，包含 CEO、CTO、Programmer、Reviewer 等角色。 | 教学实验、小型项目自动生成、多 Agent 研究。 |

### 学习重点

学习多智能体框架时，不要只看“几个 Agent 在聊天”。要重点观察：

- 谁负责分配任务？
- 任务完成的判断标准是什么？
- 中间产物是否结构化？
- 如何避免死循环？
- 如何处理某个角色输出质量差的问题？
- 是否支持人类在关键节点介入？

### 与当前仓库的关系

当前仓库已经有：

- `04-multi-agent/`：LangGraph Supervisor 模式。
- `07-crewai-intro/`：CrewAI 顺序协作。
- `08-autogen-intro/`：AutoGen 对话式代码协作。

第二阶段可以继续补：

- `11-metagpt-sop/`：研究 SOP 驱动的多 Agent。
- `15-production-agent-engineering/`：总结多 Agent 的终止、审计、评估和权限控制。

## 四、自主探索型 / 通用任务框架

### 核心隐喻

自主探索型 Agent 的核心是“目标驱动”。用户给出一个长期目标，Agent 自己生成任务列表，执行当前任务，观察结果，再反思并更新后续任务。

典型循环是：

```text
目标
  -> 生成任务
  -> 执行任务
  -> 观察结果
  -> 反思
  -> 更新任务列表
  -> 判断是否结束
```

### 主要特点

- 更强调自主性，而不是预先写死完整流程。
- 有明显的任务队列或计划列表。
- Agent 需要自己决定“下一步做什么”。
- 适合开放式调研、信息探索、自动化搜索和多步骤任务拆解。
- 风险是成本不可控、目标漂移、陷入循环或执行无价值任务。

### 代表框架与项目

| 框架 / 项目 | 主要特点 | 适用场景 |
| :--- | :--- | :--- |
| **AutoGPT** | 早期通用自主 Agent 代表，强调目标、记忆、任务执行循环。 | 自动化调研、Web 搜索、长任务探索。 |
| **BabyAGI** | 以任务队列管理为核心，展示“生成任务 -> 执行任务 -> 创建新任务”的最小模式。 | 学习自主 Agent 原理、构建轻量任务循环。 |
| **Voyager** | Minecraft 研究型 Agent，通过探索环境和编写代码形成技能库。 | 终身学习、技能复用、开放环境探索。 |

### 学习重点

学习这一类框架时，核心不是追求“完全无人值守”，而是理解自主循环为什么难：

- 初始目标如何拆成任务？
- 执行结果如何影响后续计划？
- 任务优先级如何调整？
- 什么时候停止？
- 如何避免无限循环？
- 哪些任务必须要求人类确认？
- 如何记录经验，避免下次重复探索？

### 与当前仓库的关系

当前仓库的 LangGraph ReAct 和 Self-RAG 已经有“行动 -> 观察 -> 决策”的雏形，但还没有完整的任务队列系统。

第二阶段建议新增：

- `12-autonomous-agents/`：手写一个最小 BabyAGI 风格任务循环。
- `13-skill-library-agent/`：研究 Voyager 风格技能库和长期学习。

## 五、底层编排与工具集成框架

### 核心隐喻

底层编排与工具集成框架更像“开发者工具箱”。它们不一定主打多 Agent 社会协作，而是解决更基础的问题：

- LLM 如何稳定调用工具？
- 如何连接外部 API、数据库、搜索引擎和本地文件？
- 如何管理上下文和记忆？
- 如何做 RAG、路由、重试、观测和评估？

### 主要特点

- 更适合工程师定制业务系统。
- 组件丰富，能连接多种数据源和工具。
- 可控性通常强于黑盒平台。
- 适合构建企业知识库、业务自动化流程和生产级 Agent 后端。

### 代表框架

| 框架 | 主要特点 | 适用场景 |
| :--- | :--- | :--- |
| **LangChain** | Agent 工具链生态庞大，提供 Tools、Memory、Retrievers、Chains 等组件。 | RAG、工具调用、LLM 应用集成。 |
| **LangGraph** | 用图结构显式建模状态、节点、条件边和循环，是复杂 Agent 编排的主流方案。 | 生产级工作流、多步骤 Agent、Self-RAG、多 Agent 调度。 |
| **LlamaIndex** | 更专注于数据索引、文档理解、查询引擎和 Agentic RAG。 | 企业私有知识库、高精度检索、数据中心型 Agent。 |
| **smolagents** | 倾向 Code-as-Actions，让 Agent 用代码表达行动。 | 轻量任务、代码执行型 Agent、快速实验。 |

### 学习重点

学习这一类框架时，要重点比较它们的“控制面”：

- 数据如何进入系统？
- 检索结果如何被筛选和合成？
- Agent 状态如何表达？
- 工具调用是否可控、可观测、可重试？
- 工作流是隐式对话，还是显式图结构？
- 框架更擅长 RAG、工具调用、代码执行，还是状态编排？

### 与当前仓库的关系

当前仓库已经重点覆盖：

- LangChain / LangGraph 基础。
- LangGraph Self-RAG。
- smolagents CodeAgent。

第二阶段建议新增：

- `10-llamaindex-agent/`：做一个 LlamaIndex 数据中心型 Agent，并与 `05-final-project/` 的 LangGraph Self-RAG 对比。

## 六、低代码 / 可视化 Agent 平台

### 核心隐喻

低代码 Agent 平台的核心是“产品化工作台”。它们把 Prompt、知识库、工具、工作流、插件、发布渠道和监控能力整合到一个可视化界面中。

这类平台不一定让开发者获得最高自由度，但能显著降低原型验证和业务落地门槛。

### 主要特点

- 面向产品经理、业务开发者和希望快速验证想法的工程师。
- 通常内置知识库、插件、Workflow、变量、发布渠道和日志。
- 适合快速搭建客服、知识库问答、流程自动化、内部助手。
- 风险是复杂逻辑可维护性有限，平台锁定和版本迁移成本需要评估。

### 代表平台

| 平台 | 主要特点 | 适用场景 |
| :--- | :--- | :--- |
| **Dify** | 开源 LLM 应用平台，提供 Prompt、RAG、Workflow、Agent 编排和应用发布。 | 企业内部应用、私有化部署、可视化工作流。 |
| **Coze** | 集成插件、工作流、知识库、Bot 发布和多渠道接入。 | 快速原型、插件型 Bot、业务侧 Agent 应用。 |

### 学习重点

学习低代码平台时，不要只看“拖拽很方便”。要重点评估：

- 同一个场景用代码框架和低代码平台分别怎么实现？
- 工作流复杂后是否还能清楚调试？
- 工具权限如何管理？
- 知识库更新和评估是否方便？
- 是否支持私有化部署？
- 是否能导出、迁移或版本管理？

### 与当前仓库的关系

当前仓库是代码优先，没有低代码平台实验。第二阶段可以新增：

- `14-lowcode-agent-platforms/`：用同一业务场景对比 Dify / Coze / 代码框架的实现方式。

这一阶段可以偏文档和实验报告，不一定需要大量代码。

## 八、推荐学习顺序

建议按以下顺序推进：

```text
10 LlamaIndex
  -> 12 Autonomous Agents
  -> 11 MetaGPT / SOP
  -> 13 Skill Library
  -> 14 Low-Code Platforms
  -> 15 Production Engineering
```

原因如下：

1. **先学 LlamaIndex**：它和现有 Self-RAG 最接近，能自然补齐“数据中心型 Agent”视角。
2. **再学 Autonomous Agents**：任务队列和自主循环是很多 Agent 框架背后的底层思想，手写一遍最有价值。
3. **再学 MetaGPT / SOP**：在理解自主循环之后，再看 SOP 如何约束多 Agent，会更容易理解它的工程价值。
4. **然后学 Skill Library**：这是从一次性任务走向长期学习的关键。
5. **再看低代码平台**：有了代码框架经验后，更容易评估低代码平台的边界。
6. **最后做生产工程总结**：把所有模式统一到上线标准里。

## 十、选择建议

如果目标是**企业知识库问答**，优先研究 LlamaIndex、LangGraph Self-RAG 或 Dify。

如果目标是**复杂软件开发协作**，优先研究 AutoGen、MetaGPT 和 LangGraph Supervisor。

如果目标是**自动化搜索 / 调研工具**，优先研究 AutoGPT / BabyAGI 的任务循环，再用 LangGraph 实现可控版本。

如果目标是**快速验证业务想法**，优先使用 Coze 或 Dify，但要同时评估后续迁移和可维护性。

如果目标是**生产级 Agent 系统**，不要押注单一框架。更务实的方式是：

```text
LangGraph / Workflow 负责主流程控制
LlamaIndex / Retriever 负责数据检索
专用 Agent 负责动态子任务
低代码平台负责快速交付部分业务场景
评估与观测系统负责上线治理
```

第二阶段的核心目标，就是建立这种“按问题选择模式，按场景选择框架”的判断力。
