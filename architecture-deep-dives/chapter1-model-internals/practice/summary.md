# 第一章：模型底层原理 —— 实践总结

## 1. Token 效率实战
通过 `token_analysis.py` 的测试，我们观察到：
- **模型差异**：GPT-4 (tiktoken) 在处理中文时效率较低（1字符 > 1 Token），而 Qwen2.5 (transformers) 对中文有显著优化（2.3字符/Token）。
- **对 Agent 的影响**：使用中文优化模型可以有效降低费用并变相增加 Agent 的有效记忆长度。

## 2. 上下文缓存 (Prompt Caching)
- **核心原则**：**静态在前，动态在后**。
- **最佳实践**：
    - ✅ 将 `System Prompt` 和 `Tool Descriptions` 放在请求的最前端。
    - ❌ 严禁在 Prompt 开头插入 `Session ID`、`Timestamp` 或 `Random Seed`。

## 3. 微调实验室 (Fine-tuning Lab) 实战沉淀
我们通过 Qwen2.5 (1.5B/7B) 模型，跑通了从数据构造到 SFT+DPO 的全链路。

### 3.1 模型规模与逻辑韧性
- **0.5B 模型**：极其脆弱，容易在少量微调后出现乱码或过拟合。
- **1.5B 模型**：个人开发者的“甜点位”，兼顾了下载速度与基础逻辑。
- **7B 模型**：工业级 Agent 的起点，具备真正的逻辑涌现能力。

### 3.2 2025 现代微调接口 (TRL 0.12.0+)
- **参数解耦**：最新规范要求使用 `SFTConfig` 和 `DPOConfig` 管理参数。
- **接口变更**：`SFTTrainer` 中的 `tokenizer` 已正式更名为 **`processing_class`**。
- **长度控制**：`max_seq_length` 已统一为 **`max_length`**。

### 3.3 DPO 避坑的关键：分词对齐
- **Mismatch 风险**：直接拼接字符串会导致分词对齐失败，产生训练警告。
- **解决方案**：使用**消息列表 (List of Messages)** 格式构造 DPO 数据，让库自动应用聊天模板（Chat Template），确保 Token 级对齐。

### 3.4 硬件加速 (Mac MPS)
- **MPS 优势**：在 M2/M4 Pro 芯片上，利用 `torch.device("mps")` 可获得数倍于 CPU 的训练速度。
- **内存预估**：
    - 1.5B 模型微调建议 **16G** 内存。
    - 7B 模型微调建议 **32G-48G** 内存。

## 4. 核心结论
- **SFT (有监督微调)**：负责“形”，赋予模型 Agent 的思考格式（ReAct）。
- **DPO (直接偏好优化)**：负责“神”，赋予模型是非判断力，根治死循环与幻觉。
- **LoRA (低秩适配)**：是个人开发者在本地炼制“垂直领域专家”的最强补丁。
