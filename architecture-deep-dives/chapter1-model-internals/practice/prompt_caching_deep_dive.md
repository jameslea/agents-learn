# 深度全鉴：Prompt Caching（提示词缓存）全链路指南

> **核心价值**：在 Agent 开发中，Prompt Caching 是平衡“极致性能”与“低廉成本”的唯一技术支柱。一个优化良好的缓存系统，能让多轮对话的 Token 支出降低 50%-90%，并将首字响应时间（TTFT）缩短至毫秒级。

---

## 一、 底层物理机制：前缀匹配 (Prefix Matching)

### 1. 数学上的“连续性”要求
Prompt Caching 的本质是重用计算好的 **KV Cache**。为了保证生成结果的确定性，缓存系统遵循以下铁律：
*   **锚点固定**：匹配必须从 Prompt 的 **第 1 个 Token** 开始。
*   **严苛连续**：必须是逐个 Token 的连续序列匹配。
*   **蝴蝶效应**：只要中间有一个 Token（甚至是一个空格）发生变化，该位置之后的所有 KV Cache 都会失效，无法被后续请求复用。

### 2. 逻辑演示
*   **请求 A**: `[系统指令] + [工具定义] + [问题1]`
*   **请求 B**: `[系统指令] + [工具定义] + [问题1] + [答案1] + [问题2]`
*   **缓存命中**: 请求 B 在处理时，直接从内存读取 `[系统指令] + [工具定义] + [问题1]` 的计算结果，仅需计算 `[答案1] + [问题2]`。

## 二、 厂商实现差异与技术规格

不同厂商在物理实现上（步长、触发阈值）有显著差异，这决定了你的 Prompt 长度必须超过多少才“值得”优化。

| 特性 | DeepSeek (目前最激进/领先) | OpenAI (GPT-4o) | Anthropic (Claude) |
| :--- | :--- | :--- | :--- |
| **触发方式** | **全自动** (系统后台自动识别前缀) | **全自动** (系统后台自动识别) | **显式控制** (需在消息体加 `cache_control`) |
| **最小触发长度** | **64 Tokens** (极低门槛) | 1024 Tokens | 1024 Tokens |
| **缓存步长 (Block)** | **64 Tokens** (以 64 为单位切块) | 未公开 (推测较大) | 未公开 |
| **生命周期 (TTL)** | 无活动后约 10-30 分钟失效 | 无活动后约 5-10 分钟失效 | 无活动后约 5 分钟失效 |
| **经济收益** | 缓存命中部分约 **0.1 元/1M** (极致低价) | 缓存命中部分约 **50% 折扣** | 缓存命中部分约 **90% 折扣** |

## 三、 Agent 开发中的“五大缓存杀手” (工程陷阱)

在 Agent 系统中，很多直觉上的编程习惯会瞬间摧毁缓存命中率：

### 1. 动态变量置顶
*   ❌ **错误**：`f"当前时间：{datetime.now()}\n你是一个系统助手..."`
*   **后果**：第 1 个 Token 每次都在变，万字 System Prompt 永远无法命中。
*   ✅ **修正**：将所有时间戳、随机数、Session ID 移动到 Prompt 的**最末尾**。

### 2. 用户 ID 污染
*   ❌ **错误**：`f"User-ID: {uid}\n{global_system_prompt}"`
*   **后果**：不同用户无法共享 System Prompt 缓存。若有 1000 个并发用户，服务器会重复计算 1000 次相同的背景知识。
*   ✅ **修正**：在最前面放全局共享的静态内容。

### 3. 非确定性的列表生成
*   ❌ **错误**：使用 Python 的 `set` 或无序 `dict.keys()` 生成工具描述。
*   **后果**：不同进程或不同版本下，工具出现的顺序可能随机变化，导致 Token 序列不一致。
*   ✅ **修正**：**强制排序**。所有工具描述、API 定义必须通过 `sorted()` 固化顺序。

### 4. 中间插入式 RAG (Context Jitter)
*   ❌ **错误**：在对话历史中间（如第 3 轮）强行插入检索到的背景资料。
*   **后果**：该位置之后的所有对话缓存全部作废。
*   ✅ **修正**：始终在 Prompt 的**最末尾（当前问题之前）**添加最新的 RAG 资料。

### 5. 格式微调 (Whitespace Jitter)
*   ❌ **错误**：模版引擎生成的缩进不一致，或不同版本的代码多打了一个换行符。
*   **后果**：肉眼看不见，但 Token 编码完全变了。
*   ✅ **修正**：严格管控模版，确保输出是 Token 级确定的。

## 四、 黄金架构：漏斗形 (Funnel) Prompt 构建法

为了最大化缓存收益，架构师应严格遵循“静态在前，动态在后”的层级结构：

1.  **全局静态层 (Global Static)** [命中率 > 99%]
    *   *内容*：系统角色定义、永远不变的业务法律合规约束。
    *   *收益*：跨用户、跨 Session 全球共享。

2.  **工具/技能定义层 (Skills Definition)** [命中率 > 90%]
    *   *内容*：所有可调用的 Function Calling 描述、API 规范。
    *   *收益*：只要系统不发布新功能，这部分永久固化。

3.  **用户画像/长期背景 (User Profile)** [命中率 60-80%]
    *   *内容*：该用户的姓名、历史偏好、已知的长期事实。
    *   *收益*：该用户在当天的所有对话都能复用。

4.  **历史对话摘要 (Compressed History)** [命中率 40-60%]
    *   *策略*：当对话过长时，不要无限制携带原文。使用轻量级模型生成 **摘要 (Summary)** 并替换老旧对话。
    *   *收益*：重置缓存起点，防止 Lost in the middle。

5.  **当前动态层 (Current Context & Query)** [命中率 0%]
    *   *内容*：最新的 RAG 检索结果、用户刚问的具体问题。
    *   *收益*：这是唯一需要支付“全价”计算费用的部分。

## 五、 验证、度量与优化

### 1. 监控字段
在 API 响应的 `usage` 字典中，必须严格监控以下核心指标：
*   `prompt_cache_hit_tokens`：命中数（这是你的“省钱指数”）。
*   `prompt_cache_miss_tokens`：未命中数（这是你的“优化空间”）。

### 2. KPI 指标建议
*   **基础指标**：`缓存命中率 = hit_tokens / total_prompt_tokens`。
*   **业务指标**：`单轮平均首字耗时 (Avg TTFT)`。如果命中率高，该指标应呈现断崖式下跌。

---
**架构师寄语**：在 Agent 开发中，你写的每一行 Prompt 拼接代码，其实都是在进行一次“性能与财务”的对弈。记住：**静态在前，动态在后**，这就是你的省钱秘籍。
