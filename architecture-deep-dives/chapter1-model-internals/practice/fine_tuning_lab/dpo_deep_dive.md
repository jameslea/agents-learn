# 深度专题：DPO (直接偏好优化) —— 让 Agent 拥有判断力

> **摘要**：如果说 SFT（有监督微调）是教模型“如何说话”，那么 DPO（Direct Preference Optimization）就是教模型“如何思考和选择”。在 Agent 开发中，DPO 是解决模型死循环、幻觉和拒绝服务问题的杀手锏。

---

## 一、 DPO 的本质：从“学舌”到“对齐”

在 SFT 阶段，我们给模型看的是 $[Prompt, Answer]$ 这样的一一对应关系。模型学会的是**模仿**范文的概率。
但现实中的 Agent 任务是复杂的，经常会出现多个路径：
*   **路径 A (Chosen)**：API 报错了，模型分析错误原因并告知用户。
*   **路径 B (Rejected)**：API 报错了，模型像复读机一样不停地重新调用工具。

**SFT 的局限**：它只能教模型“什么是对的”。
**DPO 的威力**：它直接告诉模型“什么是对的，且什么是错的”。它不仅增加了正确答案的权重，还主动**压制**了错误答案的生成概率。

---

## 二、 数学原理的降维理解

在 DPO 之前，人类使用 **RLHF (基于人类反馈的强化学习)**。RLHF 非常复杂，需要先训练一个“奖励模型”当教练，再通过 PPO 这种极不稳定的算法进行训练。

**DPO 的突破**：
斯坦福大学的研究者通过数学推导发现：**语言模型本身就可以充当自己的奖励模型。**
*   **公式简化**：DPO 通过计算“当前模型”相对于“参考模型”在 Chosen 和 Rejected 上的概率比值。
*   **目标**：拉大 Chosen 和 Rejected 之间的差距。
*   **结果**：不需要额外的奖励模型，不需要复杂的强化学习，只需一个交叉熵损失函数就能完成对齐。

---

## 三、 Agent 实战：DPO 能解决哪些顽疾？

### 1. 终结死循环 (Dead Loop)
当 Agent 调用的工具返回错误（如 401 Unauthorized）时，很多模型会因为 SFT 数据中“总是调用工具成功”的偏见，陷入“报错 -> 重新调用 -> 报错”的死循环。
*   **DPO 方案**：将“报错后重试”标为 `Rejected`，将“报错后向用户解释”标为 `Chosen`。

### 2. 抑制幻觉 (Hallucination)
当用户问了一个模型不知道的问题（如内部私有文档外的内容）。
*   **DPO 方案**：将“胡编乱造”标为 `Rejected`，将“诚实地回答不知道”标为 `Chosen`。

### 3. 拒绝“复读机”行为
有些模型在处理长文本时会反复输出同一句话。
*   **DPO 方案**：将重复的输出序列标为 `Rejected`。

---

## 四、 调参实战：Beta 参数的奥秘

在 `train_dpo.py` 中，你最需要关注的是 **`beta`** 参数：
*   **`beta` (通常取值 0.1 - 0.5)**：
    *   **大 beta (0.5)**：意味着你非常信任原始模型，不希望 DPO 把模型改得面目全非。模型会表现得更保守。
    *   **小 beta (0.1)**：意味着你希望模型强烈地学习你的偏好数据。如果你的偏好数据质量极高，这能让模型瞬间变聪明。

---

## 五、 总结：Agent 架构的终极形态

一个成熟的 Agent 模型生产流水线应该是：
1.  **Base Model**：提供基础智商。
2.  **SFT (有监督微调)**：注入工具调用的格式、ReAct 思考链。
3.  **DPO (直接偏好优化)**：注入业务价值观，学会分辨好坏路径，解决死循环和幻觉。

**“SFT 给模型穿上西装，DPO 教模型如何待人接物。”**
