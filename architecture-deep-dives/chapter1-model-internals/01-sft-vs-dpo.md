# 深度探索 1.1：微调技术 (Fine-tuning) —— 赋予 Agent “灵魂”与“规矩”

> **探索目标**：深入理解 SFT、DPO、PPO 的全称、原理及在 Agent 开发中的具体应用场景，明确从“提示词工程”向“模型架构调优”转型的技术路径。

---

## 一、 核心定义：什么是微调 (Fine-tuning)？
在 Agent 开发语境下，微调是指在通用大语言模型（Base Model，如 Llama-3, DeepSeek-V3, Qwen2.5）的基础上，使用特定任务的专用数据集进行二次训练。其目的是让模型从“通才”变成“专才”，在特定业务逻辑、格式遵循或思考偏好上达到生产级要求。

---

## 二、 三大核心技术详解

### 1. SFT - Supervised Fine-Tuning (监督微调)
*   **全称解释**：**Supervised** (有监督的) **Fine-Tuning** (微调)。
*   **核心原理**：这是最基础且最成熟的微调方式。它通过提供成千上万组 **“指令/问题 (Prompt) -> 标准答案 (Completion)”** 的高质量数据对，让模型通过“模仿学习”掌握特定模式。
*   **在 Agent 中的实战价值**：
    *   **格式强制 (Format Enforcement)**：如果 Agent 在处理多层嵌套 JSON 或特定的 Tool Call 语法（如 XML）时频繁报错，SFT 可以通过海量正确样本让模型形成“格式直觉”，从而在不依赖复杂 System Prompt 的情况下实现极高的格式准确率。
    *   **逻辑注入 (Chain-of-Thought Injection)**：通过提供包含完整 **Thought (思考过程)** 链条的数据（即 ReAct 范式：思考 -> 行动 -> 观察 -> 结论），训练模型学会像专业工程师一样思考，而不是直接跳到答案。
    *   **私有知识内化**：将特定的业务术语、API 文档逻辑通过 SFT 转化为模型的“原生知识”。

### 2. DPO - Direct Preference Optimization (直接偏好优化)
*   **全称解释**：**Direct** (直接) **Preference** (偏好) **Optimization** (优化)。
*   **核心原理**：由斯坦福大学团队于 2023 年提出。它颠覆了传统的 RLHF（基于人工反馈的强化学习）流程。DPO 不需要训练复杂的奖励模型（Reward Model），而是直接给模型输入成对的数据：**一个好的回答 (Chosen)** 和 **一个较差的回答 (Rejected)**。模型通过数学公式直接学习如何增加 Chosen 的概率并同时压制 Rejected 的概率。
*   **在 Agent 中的实战价值**：
    *   **行为纠偏 (Behavior Correction)**：当 Agent 出现“过度啰嗦”、“复读机”、“遇到错误就摆烂（总说对不起我无法回答）”等坏习惯时，通过 DPO 告诉模型：“这种解决问题的路径是明智的，那种是愚蠢的”。
    *   **偏好对齐 (Alignment)**：让 Agent 的决策逻辑更符合特定业务价值观。例如：在多个工具都能解决问题时，通过 DPO 训练 Agent 优先选择成本最低或速度最快的工具。

### 3. PPO - Proximal Policy Optimization (近端策略优化)
*   **全称解释**：**Proximal** (近端) **Policy** (策略) **Optimization** (优化)。
*   **核心原理**：这是一种基于强化学习（Reinforcement Learning）的算法。它需要模型在模拟环境中不断尝试（探索），并由一个“奖励模型（教练）”给出即时反馈。PPO 算法确保模型的更新步伐不会迈得太大，从而维持训练的稳定性。
*   **在 Agent 中的实战价值**：
    *   **长链条复杂规划**：当一个任务需要 Agent 连续执行 10 步以上，且中间过程难以定义“标准答案”时，PPO 可以根据任务最终的成败得分，自动回溯并优化中间的每一步决策逻辑。
    *   **缺点**：训练极度不稳定，计算资源消耗巨大，在工业界正逐渐被更轻量化的 DPO 技术取代。

## 三、 Agent 开发者的技术选型建议

| 维度 | SFT (监督微调) | DPO (直接偏好优化) | PPO (近端策略优化) |
| :--- | :--- | :--- | :--- |
| **技术难度** | 中（重点在数据清洗） | 中（重点在正负样本对构建） | 高（需要 RL 专家背景） |
| **解决的核心问题** | “听不听话？”（格式与知识） | “聪不聪明？”（逻辑与偏好） | “能不能完成极难长任务？” |
| **数据需求** | “问题 + 唯一正确答案” | “问题 + 好答案 + 坏答案” | 动态环境反馈评分 |
| **推荐程度** | **必选** (Agent 的地基) | **进阶首选** (提升智能感) | **慎选** (研发成本极高) |

## 四、 实战避坑：SFT 与 DPO 的本质差异

> **💡 架构师实战 Tips (2025年 4月更新)**：
> 
> 1. **接口规范变更**：在使用最新的 `trl` 库 (0.12.0+) 时，注意 `SFTTrainer` 的参数已由 `tokenizer` 更名为 `processing_class`，且长度参数建议统一写在 `SFTConfig` 的 `max_length` 中。
> 2. **DPO 的“分词对齐”陷阱**：在构造 DPO 数据时，强烈建议使用 **Message List** 格式（由 `prompt`, `chosen`, `rejected` 组成且内部包含 role 字典）而非拼接好的 Raw String。这能确保在计算 `logps` 时，Prompt 与 Response 之间的特殊 Token 被完美对齐，从而避免 `Mismatch warning`。
> 3. **模型规模的选择**：对于 Agent 任务，微调 1.5B 级别以上的模型才能获得较好的逻辑韧性。0.5B 及以下的小规模模型在微调后极易出现乱码或丧失对话逻辑。

很多开发者在初期会觉得 SFT 和 DPO 区别不大
（都是准备数据然后调参），PPO 成本太高不用管。但实际上，SFT 和 DPO 在**学习机制**和**数据成本**上有本质区别：

1. **模仿学习 (SFT) vs 辨别学习 (DPO)**
   - **SFT**：模型只会“模仿”标准答案（填鸭式）。如果遇到没见过的情况，容易不知所措。
   - **DPO**：模型通过“对比”学习（好 vs 坏）。它学会了避坑，知道“不该做什么”，这对解决 Agent 的死循环和幻觉尤为关键。
2. **数据成本的非对称性**
   - 构造 SFT 数据要求人类专家“从零写出完美答案”，成本极高。
   - 构造 DPO 数据只需人类专家“在模型生成的两个答案中选出更好的一项”，评价成本远低于创作成本。
3. **“教它怎么做” vs “教它别怎么做”**
   - 当 Agent 反复调用同一个工具陷入死循环时，SFT 很难教它跳出（SFT 只能教正确路径）。而 DPO 可以将死循环路径标记为 `Rejected`，模型立刻就能学会“悬崖勒马”。
