# 深度探索 1.2：Tokenizer 与上下文工程 (Tokenizer & Context Engineering)

> **探索目标**：揭开大语言模型“记忆容量”的底层秘密，解决 Agent 在处理长文本时的“失忆”问题，并掌握利用 Prompt Caching 大幅降低 API 成本的核心技术。

---

## 一、 深入底层：Tokenizer (分词器) 到底是什么？

初学者常以为模型处理的是“汉字”或“单词”，但实际上模型处理的是 **Token**。Tokenizer 就是将人类文本转化为模型能看懂的数字 ID 的“翻译官”。

### 1. Token 并不是词汇的一一对应
目前主流的大模型（如 GPT-4, Llama-3, DeepSeek）大多使用 **BPE (Byte-Pair Encoding)** 算法。
- 对于英文，一个 Token 大约等于 0.75 个单词（例如 "hamburger" 可能被切分为 "ham", "bur", "ger"）。
- 对于中文，由于大部分模型主要在英文语料上训练，中文的切分效率通常很低。可能一个汉字会被切分成 2-3 个 Token，这直接导致**相同的上下文窗口，装下的中文信息量远少于英文**。
- **实战启示**：如果你发现国产模型（如 DeepSeek、Qwen）在处理中文长文档时比 Llama-3 聪明，往往不是因为模型智商更高，而是因为国产模型的 Tokenizer 对中文进行了专门优化，压缩率更高，使得同样的 128K 窗口能塞进更多的中文背景知识。

### 2. RAG 系统中的 Tokenizer 陷阱
在编写 Agent 的 RAG（检索增强生成）模块时，很多开发者使用简单的 `text.split("\n")` 或按字符数截断文档（Chunking）。
- **风险**：如果你刚好从一个完整词汇的中间（甚至一个汉字的字节中间）截断，模型读到的就是一个“乱码” Token。
- **专业做法**：在做文本分割时，必须使用与目标大模型一致的 Tokenizer 库（如 `tiktoken` 对于 OpenAI）来进行严格的 Token 级截断。

## 二、 上下文窗口陷阱：Lost in the Middle (中间失忆)

很多模型宣称有 128K 甚至 1M 的上下文窗口，但在 Agent 场景中，这往往是一个陷阱。

- **现象**：当 Agent 经历了 20 轮对话，或者在 RAG 中塞入了 50 篇文档后，模型往往能记住文档的**开头（系统指令）**和**结尾（最近的问题）**，但会完全忽略中间的检索内容。这就是著名的 **Lost in the Middle** 效应。
- **Agent 架构优化对策**：
    1. **强制排序**：在将检索结果喂给 Agent 之前，必须使用 Rerank 模型进行重排序，将最相关的信息强行放在 Prompt 的最开头或最结尾。
    2. **滚动总结 (Sliding Window Summarization)**：不要把历史对话原文一直带在上下文中。让另一个轻量级 Agent 专门负责将旧的对话“压缩总结”为两三句话的摘要，以此替换原文。

## 三、 改变游戏规则的技术：Prompt Caching (提示词缓存)

如果你使用 LangGraph 或 ReAct 范式开发 Agent，你会发现一个巨大的痛点：**又贵又慢**。

### 1. 为什么 Agent 那么烧钱？
因为 Agent 的每一次思考循环（Thought -> Action -> Observation），都需要把**所有的历史记录、极长的 System Prompt 和几十个工具的描述**重新发送给 API。
假设系统提示词和工具描述有 3000 Token：
- 第 1 步：发送 3000 Token
- 第 2 步：发送 3000 + 500 (历史) = 3500 Token
- 第 3 步：发送 3500 + 500 (历史) = 4000 Token
绝大部分算力和成本都浪费在重复处理那 3000 Token 的静态前缀上！

### 2. Prompt Caching 的原理
目前 Anthropic (Claude)、OpenAI 和部分优秀的开源推理框架（如 vLLM）支持了提示词缓存。
- 当大模型处理文本时，它需要计算并生成 KV Cache（键值缓存）。
- **Prompt Caching** 允许你把不会变的 System Prompt 和工具说明固定下来。API 服务端会将这部分计算好的 KV Cache 储存在内存中。
- 当 Agent 进行第 2、第 3 步循环时，由于前缀匹配，模型**直接跳过这 3000 Token 的计算**。

### 3. 实战：API 层是如何实现缓存的？
很多开发者误以为需要先调一个接口去“建立缓存”，实际上主流平台的缓存机制通常是随请求自动完成的，分为两派：

- **自动前缀匹配 (OpenAI / DeepSeek)**：
  你不需要改代码。当发送请求时，系统会自动将当前请求的 Token 序列与最近几分钟内的历史请求对比。只要**前缀完全一致**（且长度达到一定阈值，如 1024 Token），就会自动命中缓存。
  - **⚠️ 致命避坑**：绝不要把时间戳、动态的 SessionID 或随机变化的短期记忆插在 System Prompt 的开头！一旦开头第一个字变了，整个长达万字的前缀匹配就会瞬间失败。动态变量必须放在 Prompt 的最末尾。
- **显式标记 (Anthropic Claude)**：
  不需要单独调接口，但需要在单次请求的 JSON 体中，为静态文本块（如工具说明）打上 `"cache_control": {"type": "ephemeral"}` 标签。系统遇到该标签会自动保留缓存，下次遇到相同带标签的文本块则直接复用。
