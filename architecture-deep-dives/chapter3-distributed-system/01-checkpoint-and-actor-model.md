# 深度探索 3.1：分布式 Agent 架构与状态护航 (Systems & Infra)

> **探索目标**：从单机跑 Demo 到生产环境跑服务，Agent 架构的复杂度会呈指数级上升。我们将探索如何通过消息队列和持久化 Checkpoint 机制，让一个长耗时、高不确定性的 Agent 在生产环境中稳定运行。

---

## 一、 认知转换：Agent 服务器 ≠ 传统 Web 服务器

传统 Web 开发（如 Spring Boot 或 Django）的信条是 **"无状态 (Stateless)"** 和 **"毫秒级响应"**。用户发一个请求，服务器查一下数据库，200 毫秒内返回。

但 Agent 彻底打破了这个规矩：
1. **超长耗时**：一个执行深度调研的 Agent，可能需要思考、调用搜索 API 10 次、爬取 5 个网页，整个过程耗时可能长达 10 分钟。
2. **极度脆弱**：在这 10 分钟内，只要遇到一次 API 超时、一次网络闪断，或者哪怕是代码抛出了一个小 Exception，如果没有任何保护机制，Agent 的进度就会瞬间清零，前功尽弃。
3. **强状态依赖**：Agent 每走一步，它的"记忆区"都在膨胀。第 5 步的决定绝对依赖于前 4 步收集到的线索。

## 二、 破局方案一：异步任务与消息队列

### 核心原则

绝不能让 Agent 逻辑在处理 HTTP 接口的同步线程里跑。这是底线，没有例外。

**生产级架构标准：**
1. **网关层接收请求**：用户点击"开始调研"，Web 服务器立刻把任务参数打包，写入消息队列，然后直接返回给前端一个 HTTP 202 (Accepted) 和一个 `Task_ID`。
2. **Worker 节点消费**：隐藏在后台的 Agent Worker 节点（可以根据负载随时横向扩容）从队列中领走任务开始"漫长"的执行。
3. **WebSocket 推送状态**：在 Worker 执行的这 10 分钟里，它不断把 `Thought (思考中...)`、`Action (正在检索...)` 通过 WebSocket 或 SSE 推送给前端。

**意义**：用户的浏览器关了也无所谓，Agent 在后台依然坚定地执行；如果流量洪峰来临，只要往集群里塞更多的 Worker 节点即可。

### 消息队列选型

| 方案 | 适用场景 | 注意事项 |
|------|---------|---------|
| RabbitMQ | 生产级任务分发 | 运维成本中等，功能完备 |
| Apache Pulsar | 超大规模、多租户 | 功能最强，运维最高 |
| Redis Streams | 中小规模，已有 Redis | 比 Redis List 可靠（支持 ACK），但持久化不如 RabbitMQ |
| **Redis List** | **原型阶段** | **生产不推荐 — 没有 ACK，Worker 崩溃丢消息** |
| Celery | Python 团队快速上线 | 封装了 RabbitMQ/Redis，提供重试、定时等抽象 |

> **不推荐用 Redis List (`RPOP`) 做生产消息队列**。没有 ACK，Worker 崩溃消息就永久丢失。必须用 Redis 的话请用 **Redis Streams**。

### 简化替代方案

单机 + 不需要扩容时，可以不引入消息队列：

```
FastAPI/ASGI → asyncio.create_task() → WebSocket 推状态 → 写入本地数据库
```

代价是放弃横向扩容能力。

## 三、 破局方案二：Checkpoint 与断点续传

这是高级 Agent 框架（如 LangGraph）的核心杀手锏。

### 核心逻辑
- Agent 被定义为有限状态机
- **每步执行完后**，状态（变量、对话历史、工具调用结果）被序列化存入数据库（PostgreSQL JSONB 或 Redis）
- **灾难恢复**：断电重启后，从数据库读出最近 Checkpoint，直接从断点继续，不需要重头推演

### Time Travel（时间旅行与人工介入）
调出旧 Checkpoint → 人工修改变量或提示词 → 沿新时间线继续运行。

**时间旅行的事实成本**：回溯重放不会回滚已发生的外部副作用（已发的邮件、已创建的订单）。这不是 Checkpoint 机制本身能解决的，需要与 Side Effect 管理配合（见第六节）。时间旅行更适合**只读场景**（审计、调试），而不是"修正历史继续跑"。

### 工程挑战

1. **差分检查点**：每步全量存储会迅速膨胀。生产环境只存增量，每隔 N 步创建新基线。
2. **清理策略**：TTL 过期（超 7 天删除）、版本链裁剪（只保留最近 K 个）、关键节点保留（只在特定节点存）。

## 四、 破局方案三：多 Agent 协调模式

多 Agent 协调没有"终局"方案，只有**模式匹配**。以下是四种常见模式：

| 模式 | 核心思想 | 适用规模 | 代表框架 |
|------|---------|---------|---------|
| **编排器** | 一个中心调度器分配任务 | 3-5 个 Agent | LangGraph、CrewAI |
| **黑板** | 共享上下文，Agent 独立读写 | 5-20 个 Agent | AutoGen（接近） |
| **Actor** | 独立私有状态 + 异步消息 | 20+ Agent | Ray、Akka |
| **消息总线** | 通过总线发布/订阅，不直接感知对方 | 动态网络 | 自建 |

> **Actor Model 并不能"防止死锁"**。它通过避免共享内存消除数据竞争，但 Agent 之间的循环依赖等待仍然会导致逻辑死锁。实际系统需要超时机制或编排层来防范。

## 五、 破局方案四：框架选型决策

| 场景 | 推荐 | 理由 |
|------|------|------|
| Python 团队，需要 Checkpoint | **LangGraph** | 内置 PostgresSaver，学习曲线最低 |
| 金融级可靠性 | **Temporal** | Durable Execution，断电毫秒级恢复，但运维成本高 |
| 快速原型，任务简单 | **Celery** 或 asyncio | 无需额外基础设施 |
| 大量 Agent 并行 | **Ray** | Actors + 分布式调度 |
| 多 Agent 对话协作 | **AutoGen** | 内置对话模式和角色定义 |
| 零运维 | **OpenAI Assistants API** | 托管式 Thread+RUN，但绑定 OpenAI |
| 多语言异构系统 | **Dapr** | 语言无关的分布式原语层 |

**关键区分**：Celery 是任务队列（崩溃后从头重试），Temporal 是 Durable Execution 引擎（从断点毫秒级恢复），两者解决的问题层次不同，不应视为对等选项。

## 六、 三个核心陷阱

### 陷阱 1：副作用重放（Replay Safety）

Agent 崩溃后从 Checkpoint 恢复，会重新执行之前的步骤，导致外部 API 被重复调用。

**应对方案**：
- **幂等性**：每次调用带上幂等键（Idempotency Key）
- **Side Effect 注册**：将外部调用剥离为"仅执行一次"的操作（Temporal Activity 有去重语义）
- **检查点后提交**：只在 Checkpoint 成功持久化后再提交副作用

### 陷阱 2：确定性约束（Determinism）

Temporal 类引擎重放时要求代码**必须产生与原始执行完全相同的输出序列**。禁止直接调用 `random`、`time.now`、外部 API，必须通过引擎提供的确定性 API 替代。

不满足的后果：重放可能选择不同的工具、调用不同的 API、产生完全不同的结果，但系统认为自己"成功恢复了"。

### 陷阱 3：外部状态漂移

恢复时外部系统状态已变化（数据库记录被修改、Token 已过期）。Checkpoint 恢复的是一段已冻结的"记忆"，但世界已经向前走了。**设计 Agent 逻辑时假设外部状态随时可能变化**——每次使用时重新获取，而不是依赖 Checkpoint 中的缓存值。

## 七、 复杂度决策树

```
你的 Agent 需要多强的保障？
│
├─ 单机，允许失败 → asyncio.run()，不需要任何基础设施
├─ 单机，不允许丢进度
│   ├─ 任务 <1 分钟 → Celery + 指数退避重试
│   └─ 任务数分钟+ → SQLite Checkpoint + 手动恢复
├─ 需要横向扩容
│   ├─ 有运维能力 → 消息队列 + LangGraph
│   └─ 快速上线 → Celery + Redis Streams
├─ 需要 Human-in-the-loop → LangGraph（PostgresSaver）或 Temporal
├─ 多 Agent 协作
│   ├─ 工作流明确（3-5）→ LangGraph / CrewAI
│   ├─ 动态网络 → AutoGen
│   └─ 大规模并行（20+）→ Ray
└─ 金融级可靠性 → Temporal
```
