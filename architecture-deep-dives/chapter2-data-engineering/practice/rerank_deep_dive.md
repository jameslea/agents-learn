# RAG 重排 (Rerank) 技术深度复盘

通过对 `benchmark_rag.py` 的调试与优化，我们对 RAG 系统中的重排环节有了更深层的工程认知。以下是本次实战的核心沉淀。

## 1. 为什么要重排？(Bi-Encoder vs. Cross-Encoder)

在基础 RAG 中，我们使用 **Bi-Encoder**（双编码器）架构。
- **特点**：文档和问题分别编码，计算余弦相似度。
- **缺陷**：缺乏 Query 与 Document 之间的**深层交叉特征**。它只能看到整体的“语义重心”，对细微的逻辑关系（如“自身免疫”对“1型糖尿病”的重要性）不敏感。

**重排 (Rerank)** 引入了 **Cross-Encoder**（交叉编码器）或 **LLM**：
- **原理**：将 Query 和 Document 同时输入模型进行联合计算。
- **优势**：模型能“逐字逐句”地对比两者的匹配度，从而纠正向量搜索中因字面相似导致的错误。

## 2. 语言模型的“水土不服”：模型选型的重要性

在实战的第一阶段，我们使用了 `ms-marco-TinyBERT-L-2-v2`，结果命中率仅为 50%。
- **学到的教训**：**RAG 组件的语言一致性是致命的。** 该模型是纯英文预训练模型，尽管 `sentence-transformers` 允许它运行，但它对中文文本的评分近乎“随机”。
- **解决方案**：转向 **LLM-as-a-Ranker**。使用 Qwen3-4B 这种具备强大推理能力的模型，能够跨越语言障碍，通过逻辑推理来判断相关性。

## 3. 性能与精度的博弈：三层检索架构

通过对比数据，我们建立了一个清晰的检索性能象限：

| 维度 | 向量检索 (Baseline) | LLM Rerank (精排) | HyDE (查询扩展) |
| :--- | :--- | :--- | :--- |
| **耗时** | 极低 (ms 级) | 高 (s 级) | 高 (s 级) |
| **擅长** | 快速过滤海量数据 | 识别逻辑混淆 | 解决提问模糊 |
| **角色** | **漏斗第一层 (召回)** | **漏斗第二层 (精滤)** | **前置增强 (补强)** |

**核心结论**：如果向量召回（Recall）层没能把正确文档捞出来（如 Top-5 里没有），重排层再强也无济于事。这就是为什么我们在实战中发现 A1 文档在 Baseline 失败时，Rerank 也束手无策。

## 4. LLM-as-a-Ranker 的工程技巧

在使用 Qwen 作为重排器时，我们采用了以下最佳实践：
- **Zero-shot Scoring**：通过 Prompt 让 LLM 给出 0-10 的评分，而非简单的“是/否”。这提供了更细粒度的排序依据。
- **Prompt 设计**：要求模型“仅输出数字”，并配合正则清洗。这保证了在 Agent 自动化流程中的稳定性。
- **Top-K 截断**：由于 LLM 耗时较高，我们通常只对召回层的前 5-10 条文档进行重排，以在精度和用户体验间取得平衡。

## 5. 对后续工作的启示

1. **HyDE + Rerank 是黄金搭档**：HyDE 负责把“针”先捞到碗里（提高召回），Rerank 负责把这根“针”精准地挑出来（提高精度）。
2. **本地模型链条化**：利用 LM Studio 提供的兼容接口，我们可以轻松地将 Embedding、Chat、Rerank 串联在一个本地闭环内，这对于隐私敏感或成本敏感的场景至关重要。
3. **下一步方向**：当文档间的关系不再是简单的语义相似，而是复杂的逻辑网络时，简单的 Rerank 也会遇到瓶颈，这就是为什么要引入 **GraphRAG**。
