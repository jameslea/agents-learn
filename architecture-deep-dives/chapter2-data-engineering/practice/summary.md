# 第二章：深度数据工程 —— 实战总结

## 1. 高级检索策略对比
在本章实战中，我们对比了三种 RAG 模式在处理“语义混淆”数据集时的表现：

### 1.1 Baseline (向量检索)
- **原理**：直接将 Query 转换为向量，在库中进行余弦相似度搜索。
- **表现**：容易受到“字面相似度”干扰。例如搜索“苹果生病”，可能返回关于 iPhone 故障或苹果公司的新闻，而非苹果树病害。

### 1.2 Rerank (重排序)
- **原理**：先用向量检索捞出 Top-N，再用模型对 Query 和 文档对进行逐一打分。
- **表现**：
    - 使用小型 Cross-Encoder (如 ms-marco-TinyBERT) 处理中文时表现不佳，容易出现漏删正确文档的情况。
    - **进阶实战**：使用 **LLM Reranker (Qwen3-4B)** 作为重排器，命中率从 50% 显著提升至 83.3%。LLM 具备极强的语义理解能力，能精准识别医学和法律领域的深层关联，但耗时显著增加（平均 ~3s/请求）。

### 1.3 HyDE (查询扩展)
- **原理**：让 LLM 先写一个“假设性答案”，用答案去搜答案。
- **表现**：**稳定性最强**。在本次测试中达到了 100% 的命中率。它不仅解决了语义模糊问题，还能通过 LLM 的常识联想（如将“美国首都”扩展为“华盛顿哥伦比亚特区”）来桥接 Query 与文档之间的 Gap。

## 2. 工程最佳实践
- **LM Studio 整合**：利用本地部署的 LLM (如 Qwen) 提供的 OpenAI 兼容接口，可以零成本实现 HyDE、LLM Rerank 和 Embedding。
- **模型选择**：
    - **Embedding**: `nomic-embed-text` 在本地运行效率极高。
    - **Reranker**: 
        - 追求速度：`bge-reranker-base`。
        - 追求精度：**LLM-as-a-Ranker** (如 Qwen3-4B)，利用 LLM 的推理能力进行精排。

## 3. 核心结论
- **数据工程决定下限**：如果 ETL 解析出的文本是乱码，再强的 Rerank 也无济于事。
- **检索算法提升上限**：Rerank + HyDE 的组合可以解决 80% 以上的“搜不到”或“搜不准”问题。
