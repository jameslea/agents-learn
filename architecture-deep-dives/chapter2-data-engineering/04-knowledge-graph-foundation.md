# 深度探索 2.4：知识图谱基础与 GraphRAG 工程实践

> **探索目标**：系统理解知识图谱的核心概念、生产级构建流程与测试体系，为 GraphRAG 实战奠定理论基础。

---

## 一、知识图谱核心概念

知识图谱是用**图结构来表达人类知识**的数据模型，包含三个核心要素：

| 要素 | 含义 | 示例 |
|------|------|------|
| **实体 (Entity/Node)** | 真实世界的事物 | `特斯拉`, `Elon Musk`, `Model 3` |
| **关系 (Relation/Edge)** | 实体之间的语义连接 | `(Elon Musk) — [CEO_of] → (特斯拉)` |
| **属性 (Property)** | 实体的特征描述 | `特斯拉: {成立时间: 2003, 总部: 奥斯汀}` |

### 与传统 RAG 的本质区别

传统 RAG 的文档片段：
> "特斯拉在 2024 年 Q1 交付了 38.7 万辆汽车，低于市场预期。"

知识图谱则将同一信息拆解为确定性拓扑结构：

```
(特斯拉) — [季度交付] → (2024Q1: 38.7万辆)
(特斯拉) — [市场表现] → (低于预期)
(特斯拉) — [所属行业] → (新能源汽车)
(新能源汽车) — [竞争对手] → (比亚迪)
```

向量检索依靠词向量空间的概率分布"猜测"语义，知识图谱依靠显式的拓扑结构"断言"事实。前者是概率性的，后者是确定性的。

## 二、知识图谱的关键技术点

### 2.1 Schema 设计（决定上限）

知识图谱不是自动生成的。你需要定义本体（Ontology）：

```
实体类型: 公司 | 人物 | 产品 | 事件 | 技术
关系类型: 收购 | 投资 | 发布 | 竞争 | 合作
```

Schema 的粒度直接影响后续的查询能力：
- **太粗**（只有"关联"一种关系）：图谱退化为无意义的网，无法支持精细化查询
- **太细**（定义上百种关系）：抽取成本飙升，且 LLM 难以一致遵守

### 2.2 实体抽取精度（Garbage in, Garbage out）

Indexing 阶段的最大瓶颈来自三个层面：

| 问题 | 描述 | 影响 |
|------|------|------|
| **漏抽 (False Negative)** | 关键实体未被识别 | 关系链断裂，检索空间不完整 |
| **误抽 (False Positive)** | 普通名词被当成实体 | 图谱充满噪音，降低检索精度 |
| **指代消解 (Coreference Resolution)** | 代词与指代对象未关联 | "苹果发布了新款 iPhone，**它**的股价随即上涨"中的"它"与"苹果公司"丢失连接 |

### 2.3 实体对齐 / 去重（Entity Resolution）

同一实体在不同文档中可能有不同表述：`"苹果"`、`"Apple Inc."`、`"AAPL"`、`"苹果公司"`。未对齐的情况下，图谱中出现多个节点代表同一事物，社区检测和图算法都会产生偏差。

常见对齐策略：
- **规则匹配**：别名表、正则映射（适用于枚举值有限的场景）
- **向量相似度**：通过实体名称/描述的 Embedding 做聚类
- **LLM 辅助判定**：让模型判断两个实体是否指向同一真实事物

### 2.4 图算法选型

图谱建好后，利用其拓扑结构的常见算法：

| 算法 | 用途 | 场景 |
|------|------|------|
| **Node2Vec** | 将节点映射到向量空间 | 实体相似度搜索 |
| **Leiden / Louvain** | 社区检测，发现实体群组 | 全局摘要、话题聚类 |
| **PageRank / Personalized PageRank** | 衡量节点重要性 | 关键实体发现、推荐 |
| **图神经网络 (GNN)** | 在图结构上做预测 | 关系推理、链路预测 |

### 2.5 存储层选择

| 存储方案 | 适用场景 | 部署成本 |
|---------|---------|---------|
| **Neo4j**（原生图数据库） | 复杂图遍历、深度关系查询 | 高 |
| **NebulaGraph**（分布式） | 大规模图谱、水平扩展 | 高 |
| **NetworkX + JSON 序列化** | 轻量级、原型验证 | 极低 |
| **RDF 三元组库（如 Apache Jena）** | 语义网/W3C 标准场景 | 中 |

## 三、框架对比

### 端到端 RAG + 图谱方案

| 框架 | Schema 策略 | 本地模型支持 | 上手成本 | 定位 |
|------|-----------|------------|---------|------|
| **Microsoft GraphRAG** | LLM 自动提取 + Schema 提示示例 | 不支持（需 OpenAI） | 高 | 生产级全局摘要 |
| **LightRAG** | LLM 完全自由提取 | 支持（Ollama） | 低 | 轻量快速验证 |
| **Nano-GraphRAG** | 同 LightRAG 简化版 | 支持 | 极低 | 教学学习 |
| **Fast GraphRAG** | LLM 自动提取 | 支持 | 低 | Python 轻量化实现 |

### 专业知识图谱构建

| 框架 | 特点 |
|------|------|
| **Neo4j + LLM Graph Builder** | Neo4j 官方方案，用 LLM 从非结构化数据自动建图 |
| **Diffbot** | 商业级知识图谱提取 API |
| **DBpedia Spotlight** | 将文本链接到 DBpedia 知识库 |
| **spaCy + custom NER** | 传统 NLP 方式构建轻量图谱 |

### 图计算框架

| 框架 | 定位 |
|------|------|
| **NetworkX** | Python 标准图计算库，适合原型验证 |
| **PyG (PyTorch Geometric)** | 图神经网络框架，需 GPU |
| **DGL (Deep Graph Library)** | 另一主流 GNN 框架 |
| **igraph** | 高效 C 核心图算法库，Python 封装 |

### Schema 策略的核心差异

不同框架在 Schema 上存在根本性设计分歧：

| | 传统知识图谱 | LightRAG | Microsoft GraphRAG |
|--|-----------|----------|-------------------|
| **Schema 定义** | 手动定义 | 隐式自动生成 | 自动提取 + 提示引导 |
| **一致性** | 强 | 弱 | 中 |
| **上手成本** | 高 | 零 | 高 |
| **查询控制** | 精确 | 模糊 | 中等 |
| **跨文档一致性** | 好 | 差（需合并补偿） | 中 |

LightRAG 通过两个机制补偿一致性问题：
1. **实体合并（Entity Merging）**：插入新文档时，同名或描述相似的实体自动合并
2. **双通道检索（Dual-Level Retrieval）**：底层精确匹配实体名，高层用文本摘要做主题匹配，降低对实体类型规范化的依赖

## 四、知识图谱生产流程与测试体系

生产级知识图谱不能只靠"抽了就建"，必须建立完整的测试闭环。

### 4.1 整体流程

```
Golden Dataset 标注（人工标注 50-100 篇典型文档的实体和关系）
       ↓
Prompt / 模型调优
       ↓
抽取精度测试 ──→ 是否达标？
  (精确率/召回率/F1)    │
       ↓ 否            ↓ 是
  调整 Prompt        Schema 归一化与实体对齐规则制定
                          ↓
                     端到端检索评测（对比 Baseline）
                          ↓
                    效果优于 Baseline？──→ 部署
```

### 4.2 抽取精度测试（单元测试）

构建小规模**黄金标注集**（Golden Dataset）——从业务文档中精选 50-100 篇典型样本，人工标注所有实体和关系作为 Ground Truth。

```
召回率 = 系统正确抽出的实体数 / 标注集中的实体总数
精确率 = 系统正确抽出的实体数 / 系统抽出的实体总数
F1    = 2 * (精确率 * 召回率) / (精确率 + 召回率)
```

这里存在核心的 trade-off：
- **Prompt 宽松**（"尽量多抽"）：召回率 ↑，精确率 ↓，噪音增加
- **Prompt 严格**（"只抽确定无疑的"）：精确率 ↑，召回率 ↓，遗漏风险

没有 Golden Dataset，这个 trade-off 就无法量化，调 Prompt 只能靠感觉。

### 4.3 Schema 一致性测试（回归测试）

自动抽取的最大隐患是实体类型在不同批次中"分裂"：

```
上周抽出的类型: ["公司", "人物", "技术", "产品"]
本周抽出的类型: ["公司", "人物", "Technology", "Product", "团队"]
                              ↑ 混入英文，类型体系漂移
```

CI 中应加入 Schema 一致性检查。出现分裂时通过 Entity Resolution 层做**类型归一化映射**。

### 4.4 端到端检索评测

知识图谱最终的价值在于检索质量。测试集应覆盖不同类型的查询：

```
查询样例:
  - "特斯拉 2024 年的交付量趋势如何？"
  - "电池技术的主要分歧点有哪些？"
  - "Elon Musk 和哪些公司有关联？"

评估维度:
  - 检索到的节点 / 社区是否覆盖关键信息
  - 最终回答相比纯向量 RAG 是否有提升
  - 多跳推理（Multi-hop）场景的正确率
```

## 五、知识图谱在 Agent 开发中的难度定位

知识图谱在 Agent 体系中属于**高难度但高回报**的组件：

### 难度来源

1. **技术栈交叉**：它不是单一技术，而是 NLP + 图算法 + 存储 + Prompt Engineering + 评测的交叉产物。各层串联依赖——抽取不准，下游全废。

2. **非标准化**：相比向量检索（Embedding + FAISS + Top-K 已成定式）和 Rerank（Cross-Encoder 是标准方案），知识图谱从 Schema 设计到存储选型**每个环节都需要业务判断，没有标准答案**。

3. **投入产出比不直观**：
   - 调 Rerank 一周，Hit Rate 从 50% → 80%，效果立竿见影
   - 搭知识图谱一周，可能结论是"和简单检索差不多"
   - 知识图谱的核心价值在"连接线索"类问题（跨文档、多跳推理），在测试集中可能只占 20%

### 价值所在

大部分 Agent 团队停留在 `Embedding → FAISS → LLM` 流水线上，面对跨文档、多跳推理问题力不从心。知识图谱是当前少数能真正突破这一限制的已知手段。能做好知识图谱的团队，在架构决策上就多了一个关键维度。

## 六、推荐实战路径

基于 M4 Pro + Ollama 本地环境，建议的渐进式学习路径：

1. **LightRAG 本地部署**：用 Ollama + Qwen3 跑通最小闭环，观察实体抽取和社区识别的实际效果
2. **接入现有评测框架**：将 LightRAG 检索结果加入 `benchmark_rag.py`，量化对比 Baseline / Rerank / HyDE / GraphRAG
3. **Schema 干预实验**：尝试不同的 Prompt 策略对抽取精度的影响，建立"Schema 越精细是否效果越好"的直觉判断
4. **多跳推理测试**：构造需要连接多个实体才能回答的问题，观察 GraphRAG 在此类场景中的相对优势
