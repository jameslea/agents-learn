# MetaGPT 与 SOP 驱动的多智能体

> 本文是阶段 11 的概念预习文档。我们将探讨 MetaGPT 的核心哲学：如何将软件工程的 SOP（标准作业程序）引入 AI 智能体协作。

---

## 一、 MetaGPT 的核心理念

MetaGPT 的 Slogan 是：**"Code = SOP(Team)"**。

它的核心假设是：如果我们要让 AI 完成复杂的任务（如开发一个软件），简单的对话是不够的。我们需要把 AI 组织成一个“虚拟公司”，并为每个职位规定极其严格的产出物标准。

---

## 二、 关键组件：Role (角色)

在 MetaGPT 中，一个角色不仅是一个 Prompt，它由以下几个要素组成：

1.  **Name / Profile**：你是谁（如 Product Manager）。
2.  **Goal**：你的核心目标是什么。
3.  **Constraints**：你的行为约束。
4.  **Actions**：你能做哪些具体的事（如编写 PRD）。
5.  **Watch**：你关注哪些信息的更新。

---

## 三、 SOP：结构化的产物链

MetaGPT 最强大的地方在于它规定了信息流动的**标准格式**。Agent 之间传递的不是“聊天记录”，而是“文档”：

1.  **Product Manager** -> 产出 **PRD (Product Requirement Document)**。
2.  **Architect** -> 产出 **System Design** (包含接口、数据结构、架构图)。
3.  **Project Manager** -> 产出 **Task List**。
4.  **Engineer** -> 产出 **Code**。
5.  **QA** -> 产出 **Test Report**。

这种模式确保了：**上游的产出是下游的输入，且输入必须是结构化、可解析的。**

---

## 四、 为什么 SOP 能减少幻觉？

1.  **强制思考**：Agent 在写代码前，必须先写设计文档。这强迫 LLM 在执行前先进行高层次的逻辑建模。
2.  **明确边界**：每个角色只负责自己的一小块工作。职责越明确，LLM 越不容易出错。
3.  **审查机制**：SOP 中天然包含了 Review 环节，错误在早期（文档阶段）就能被发现，而不是等代码跑不通了再去 debug。

---

## 五、 阶段 11 学习目标

1.  **模拟 SOP 流程**：不一定要安装笨重的 MetaGPT 库，我们可以手写一个“需求 -> 设计 -> 实现”的简化 SOP 流。
2.  **理解结构化产物**：学习如何让 Agent 输出 Markdown 格式的技术文档，并作为下一个 Agent 的 Context。
3.  **对比分析**：
    *   **AutoGen**：偏向于自由对话、动态博弈。
    *   **CrewAI**：偏向于任务驱动、职场管理。
    *   **MetaGPT**：偏向于流程驱动、工程化产物。
