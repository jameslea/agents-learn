# 低代码 Agent 平台：从“手写”到“编排”

> 本阶段探讨 Dify、Coze 等可视化平台如何重构 Agent 的开发流程。通过底层逻辑的沉淀，我们现在能更清晰地看透这些“黑盒”工具。

---

## 一、 核心隐喻：可视化工作流 (DAG)

低代码平台的本质是将复杂的代码逻辑抽象为 **DAG (有向无环图)**：
*   **节点 (Node)**：代表一个具体的功能单元（LLM 调用、知识库检索、代码执行、条件判断）。
*   **连线 (Edge)**：代表数据的流动路径。
*   **变量 (Variable)**：在节点之间传递的上下文信息。

---

## 二、 核心组件与代码的对应关系

| 可视化组件 | 对应我们之前学的代码概念 |
| :--- | :--- |
| **知识库 (Knowledge)** | `LlamaIndex` 的 Index / Vector Store |
| **工作流 (Workflow)** | `LangGraph` 的 Nodes & Edges |
| **工具 (Tools)** | `LangChain` 的 Tool / Function Calling |
| **提示词编排 (Prompt Eng)** | `ChatPromptTemplate` |
| **变量管理 (Variables)** | `Pydantic` 模型与上下文传递 |

---

## 三、 Dify：生产级低代码平台的标杆

Dify 之所以受欢迎，是因为它解决了 Agent 落地中最繁琐的 **“最后三公里”**：

1.  **数据的“脏活”**：自动处理不同格式文档的清洗、切分、重排。
2.  **API 的“重活”**：提供标准的 API 接口，直接集成到现有业务系统。
3.  **观测的“累活”**：内置日志记录、Token 消耗统计、用户反馈收集。

---

## 四、 深度思考：低代码的边界

作为架构师，你需要知道低代码平台的**局限性**：

1.  **逻辑天花板**：当业务逻辑极其复杂（如需要动态递归、复杂的并发控制）时，可视化连线会变成“乱麻”。
2.  **厂商锁定**：你的逻辑高度依赖平台的组件，很难无缝迁移。
3.  **调试深度**：当模型报错时，低代码平台往往难以给出极细维度的 Trace（追踪）。
