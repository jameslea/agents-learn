- **Self-RAG 闭环**: 实现了具备”自我评估、自我改写、自我检索”能力的 RAG 系统。

- **评估与安全防御**: LLM-as-a-Judge 自动化评测（忠实度/相关性/工具成功率）、Prompt Injection 攻击手法演进、最小权限原则、多层 Guardrails 纵深防御。

## 3. 代码执行机制深度辨析 (Code Execution)

智能体”干活”的底层逻辑各异，主要分为两大流派：

- **模拟派 (AST Interpretation)** - 代表：`smolagents`
    - **原理**：在内存中解析 Python 语法树并手动模拟运行。
    - **特性**：极度安全（不接触系统资源），但功能受限（仅限白名单语法）。
- **实战派 (Process-based)** - 代表：`AutoGen`
    - **原理**：将代码落盘并启动真实的 OS 进程执行。
    - **特性**：功能全整（支持所有库），通过 Docker 实现物理隔离，具备真实的报错反馈。

## 5. 检索增强检索的工程启示 (来自 Chapter 2 实战)

在语义混淆 + 多跳推理测试集上对比了 6 种检索模式后的核心发现：

| 场景 | 推荐方案 | 命中率 |
|------|---------|-------|
| 同义/近义检索 | Baseline | 83% (0.02s) |
| 语义混淆（一词多义） | HyDE / HyDE+Rerank | 100% |
| 多跳推理（跨文档连接） | HyDE+Rerank | 88% |
| 全局总结 | GraphRAG-Global | 唯一可行方案 |
| 高召回需求 | HyDE+Rerank | 平均 94% |

**没有银弹**：HyDE 在多跳场景从 100% 降至 62%，GraphRAG-Local 在语义混淆场景仅 50%。方案选择取决于你的数据特征和延迟要求。

## 6. 分布式 Agent 的关键认知 (来自 Chapter 3)

生产级 Agent 与单机 Demo 的最大差异：

- **异步解耦是底线**：Agent 绝不能跑在 HTTP 同步线程里。单机用 asyncio，多机用消息队列。
- **Checkpoint 不是存档**：恢复的真正难点是副作用重放（支付接口会被调两次）和外部状态漂移（恢复时世界已改变）。幂等键和 Side Effect 隔离是必选项。
- **Actor Model 不防死锁**：它只消除共享内存竞争，循环依赖导致的逻辑死锁仍需超时或编排层防范。
- **按需引入复杂度**：单机 asyncio → SQLite Checkpoint → 消息队列 + LangGraph → Temporal，逐级递进。

## 7. Agent 安全的第一原则 (来自 Chapter 4)

**最小权限原则比任何 Guardrails 都重要。** 一个没有”发送邮件”工具的 Agent，即使被注入攻击，也无法发送邮件。在 Web 安全里攻击代码不会被执行除非有漏洞；在 Agent 安全里，攻击指令会**默认被执行**，因为 Agent 的职责就是按照文本指令做事。
